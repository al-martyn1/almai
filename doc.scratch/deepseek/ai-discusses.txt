# Задание на разработку утилиты `almai`

## Твоя роль
Ты — опытный Python-разработчик и архитектор программного обеспечения. Твоя задача — написать качественную, модульную и хорошо документированную утилиту для сборки препромптов из ID в единый Markdown-документ.

## Цель
Разработать скрипт на Python (название проекта — `almai`), который по заданным идентификаторам собирает препромпты из различных источников, обрабатывает зависимости, учитывает локализацию и формирует итоговый документ. Утилита должна поддерживать гибкую конфигурацию через файлы и командную строку, а также предоставлять список доступных препромптов.

## Основные функциональные требования
1. **Поиск и загрузка препромптов**  
   - Препромпты хранятся в каталогах, структура которых строго определена (подкаталоги по типам: `roles`, `objectives`, `contexts`, `formats`, `extras`, `constraints` и т.д.).  
   - Поддерживаются локализованные версии в подкаталогах с точкой и двухбуквенным кодом языка (например, `.ru`).  
   - Поиск осуществляется по нескольким источникам в порядке приоритета (командная строка → проектные каталоги → пользовательские → системные → встроенные).  
   - Учитывается текущий язык (задаётся через `--lang`, `almai.yaml` или `--default-lang`).

2. **Обработка зависимостей**  
   - Препромпты могут содержать YAML frontmatter с полем `requires`, описывающим зависимости (логические выражения И/ИЛИ).  
   - Необходимо разрешать зависимости, добавляя недостающие препромпты, проверять циклы и выдавать ошибки.  
   - Поддерживаются зависимости как внутри одного типа, так и между разными типами.

3. **Сборка итогового документа**  
   - Препромпты группируются по типам в порядке, заданном опцией `--preprompt-types`.  
   - Простые препромпты (без заголовков) объединяются в ненумерованный список (с отступами для вложенных списков).  
   - Расширенные препромпты (с заголовками) вставляются с понижением уровня заголовков на один.  
   - Добавляются файлы описания проекта (например, `PROJECT.md`) по правилам расширенных препромптов.

4. **Поддержка динамических типов препромптов**  
   - Пользователь может добавлять новые типы через `--preprompt-types`.  
   - Для каждого типа автоматически создаются опции командной строки (единственное/множественное число) и опции для текстового ввода.

5. **Команда `--list`**  
   - Выводит список доступных препромптов с учётом текущего языка и путей поиска.  
   - Формат вывода — таблица с идентификатором и описанием (из frontmatter).

6. **Подсчёт токенов**  
   - Эвристический алгоритм (по правилам, описанным в ТЗ) плюс возможность использования внешних библиотек (`tokenizers`, `rs-bpe`, `smolbpe`).  
   - Результат выводится в stderr.

7. **Обработка ошибок**  
   - Два режима: обычный (предупреждения, продолжение работы) и строгий (`--strict`).  
   - Коды возврата: 0 — успех, 1 — ошибка.

## Технические требования
- **Язык реализации**: Python 3.7+ (максимальная совместимость).  
- **Сторонние библиотеки**:  
  - Обязательно: `pyyaml` (для парсинга frontmatter).  
  - Опционально: `tokenizers`, `rs-bpe`, `smolbpe` для подсчёта токенов.  
  - Для тестов: `pytest`.  
- **Архитектура**: модульная, с чёткими интерфейсами. Рекомендуется выделить классы/модули:  
  - `ConfigManager` — загрузка конфигураций (файлы, CLI, макросы).  
  - `PrepromptLoader` — поиск и чтение файлов с учётом локализации и приоритетов.  
  - `DependencyResolver` — разрешение зависимостей.  
  - `PromptAssembler` — сборка итогового Markdown.  
  - `TokenCounter` — подсчёт токенов.  
  - `CLI` — обработка аргументов (argparse или click).  
- **Платформенная совместимость**: Windows, Linux, Unix-подобные (FreeBSD). Использовать `pathlib` для путей, встроенные средства для определения домашнего каталога.  
- **Кодировки**: все файлы в UTF-8; корректная обработка UTF-8 BOM. Вывод без BOM.  
- **Тестирование**:  
  - Набор тестов на `pytest`, покрывающих ключевую функциональность: загрузка, парсинг frontmatter, разрешение зависимостей, локализация, сборка, CLI, обработка ошибок.  
  - Тесты должны быть изолированы (использовать временные каталоги `tmp_path`).  
  - Рекомендуемая структура тестов описана в ТЗ.

## Детали реализации, требующие особого внимания
1. **Приоритет путей поиска препромптов**  
   - Пути добавляются в следующем порядке (последний имеет наивысший приоритет):  
     1. Встроенные конфигурационные файлы (основной, системный, пользовательский).  
     2. Автоматически добавленные проектные пути (`.preprompts.almai` или `.preprompts`).  
     3. Пути из командной строки (включая @-файлы).  
   - Поиск файла осуществляется в обратном порядке добавления.

2. **Обработка простых препромптов**  
   - Простые препромпты не должны содержать блоков кода. Если блок кода обнаружен, препромпт следует считать расширенным (или выдавать предупреждение).  
   - При объединении нескольких простых препромптов: удалить лишние пустые строки, добавить префикс `- ` к первой строке, к последующим строкам добавить два пробела (для вложенности).  

3. **Поле `requires`**  
   - Поддерживаются строки вида `"cpp-dev|c-dev, java-dev"`, которые преобразуются в логическое выражение: (cpp-dev OR c-dev) AND java-dev.  
   - Также поддерживается YAML-список.  
   - При неудовлетворённой OR-зависимости подключается первая существующая альтернатива.  
   - Циклические зависимости — фатальная ошибка.  

4. **Локализация заголовков**  
   - Заголовки разделов (например, "Твоя роль") задаются через `--translation`.  
   - Если перевод не найден, используется английское название типа с заглавной буквы (например, "Roles").  

5. **Подсчёт токенов**  
   - Эвристический алгоритм должен строго следовать описанию в ТЗ (особенно правила для дефиса и апострофа).  
   - При наличии внешних библиотек использовать первую доступную, с параметрами по умолчанию (для `tokenizers` — модель `"gpt2"`).  
   - Результат выводить в формате: `[heuristic token count: N]` или `[tokenizer <имя>: N]`.  

6. **Макросы в конфигурационных файлах**  
   - Поддерживаются макросы: `$(AppName)`, `$(AppRoot)`, `$(AppBin)`, `$(AppConf)`, `$(UserHome)`, `$(UserAppPath)`, `$(CWD)`.  
   - Макрос `$(ProjectRoot)` не используется в конфигах, так как на момент их обработки проект ещё не найден.  

7. **Динамические опции**  
   - Для каждого типа из `--preprompt-types` должны создаваться опции:  
     - `--тип` / `--типы` (например, `--role`, `--roles`).  
     - `--тип-text` (например, `--role-text`).  
   - Правила преобразования единственного/множественного числа описаны в ТЗ.  

## Ожидаемый результат
1. **Код программы** в виде Python-пакета с точкой входа (`almai`).  
2. **Модульная структура**, разделённая на файлы (например, `almai/__init__.py`, `almai/cli.py`, `almai/config.py`, `almai/loader.py`, `almai/deps.py`, `almai/assembler.py`, `almai/tokenizer.py`).  
3. **Дистрибутив**, готовый к установке через `pip` (pyproject.toml с зависимостями и точкой входа).  
4. **Набор тестов** на `pytest`, покрывающих функциональность (не менее 80% кода).  
5. **Документация**:  
   - Краткое описание установки и использования в `README.md`.
