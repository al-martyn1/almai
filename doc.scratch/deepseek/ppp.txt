Я хочу создать библиотеку препромптов для ИИ.
Я занимаюсь разработкой программного обеспечения на языках C++, C, Python, JS и TS и препромпты должны уточнять мои запросы к ИИ.

Также я занимаюсь маркетингом и исследованием рынка программного обеспечения, а также продажей собственного ПО, для этого мне тоже нужны типы препромптов и препромты.

Возможные типы препромптов:
- roles - твои роли
- objectives - цели проекта
- contexts - контексты задачь
- formats - выходные форматы
- extras - дополнительные уточнения
- constraints - ограничения

Придумай, какие ещё типы препромптов могут быть? Используй везде множественное число для типа препромпта (но не забудь указать и единственное, особенно, если преобразование между ними нетривиальное)

После этого придумай, какие препромпты могут быть.
Например, роли (roles) могут быть такими:
- cpp-dev - разработчик на C++ (препромпт: "опытный разработчик на языке C++");
- py-dev - разработчик на Python (препромпт: "опытный разработчик на языке Python");
- software-architect (sw-architect) - архитектор программного обеспечения (препромпт: "опытный архитектор программного обеспечения").
- devops - специалист по автоматизации (препромпт: "devops - специалист по автоматизации сборки, тестирования и развёртывания (обновления) программ");
- cmake - знаток CMake (препромпт: "ты эксперт по CMake, знаешь все тонкости написания CMakeLists.txt").

Ограничения (constraints), например, могут быть такими:
- cpp17 (препромпт: "используй C++ 17");
- py2 (препромпт: "используй Python версии 2").

Сгенерируй препромты в виде отдельных markdown файлов, считая, что они находятся в в одном каталоге с подкаталогами. 
Так, препромпт для cpp-dev должен располагаться в файле `roles/cpp-dev.md`

Правила оформления ответа в формате markdown с несколькими "прикреплёнными" файлами:

1. Каждый файл (листинг) оформляй как блок кода в Markdown.  
2. Перед блоком кода укажи имя файла (или несколько имён, каждое на новой строке). Для выделения используй **жирный**, __жирный__ или ~~зачёркнутый~~ текст.  
3. Отделяй имя файла от предыдущего содержимого пустой строкой. После блока кода также ставь пустую строку перед следующим именем.  
4. Выбирай маркеры блока кода (```, ~~~, ```` и т.п.) так, чтобы они **не совпадали** с последовательностями внутри кода. При конфликте увеличивай количество символов (4 и более) или меняй тип (бэктики <-> тильды).  
5. Если один код относится к нескольким файлам, перечисли все имена построчно перед блоком.
6. В начале приведи полностью свой ответ, и только потом "прикрепляй" файлы к документу. Между файлами не должно быть текста с объяснениями.
7. Если в ответе присутствуют блоки кода, не являющиеся результирующими файлами целиком, а лишь поясняющие какие-то либо аспекты ответа, им тоже следует задать имя файла, отражающее суть примера, файл должен располагаться в отдельном каталоге `examples`.
