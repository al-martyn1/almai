# Техническое задание на разработку утилиты `almai`

## 1. Общее описание

Утилита `almai` (AI Prompt Assembler) предназначена для сборки единого Markdown-документа, содержащего препромпты (preprompts) для систем искусственного интеллекта. Препромпты группируются по типам (роли, цели, контексты, форматы вывода, дополнительные указания, ограничения) и комбинируются с описанием проекта (файлы `PROJECT.md`, `REQUIREMENTS.md` и т.п.) в заданном порядке.

Итоговый документ может использоваться как входной промпт для LLM (например, ChatGPT, Claude) при выполнении задач, связанных с разработкой, ревью кода, проектированием и т.д.

Утилита написана на Python, должна работать в Windows, Linux и других Unix-подобных ОС.

---

## 2. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Препромпт (preprompt)** | Текст (в формате Markdown), который добавляется к финальному запросу перед основной задачей. Препромпты могут задавать роль ИИ, контекст, ограничения и т.п. |
| **Тип препромпта (preprompt type)** | Категория, к которой относится препромпт (например, `roles`, `objectives`, `constraints`). Каждому типу соответствует отдельный подкаталог в хранилище препромптов. |
| **Идентификатор препромпта (preprompt ID)** | Уникальное имя файла препромпта без расширения `.md`, используемое для ссылки на конкретный препромпт (например, `cpp-dev`). |
| **Категория препромпта** | Синоним типа препромпта. |
| **Локализация (localization)** | Возможность хранить версии препромптов для разных языков в подкаталогах вида `.ru`, `.en` и т.п. |
| **Расширенный препромпт** | Препромпт, содержащий структурные элементы Markdown: заголовки, листинги, таблицы и т.д. При вставке в итоговый документ уровень заголовков понижается на 1. |
| **Простой препромпт** | Препромпт без заголовков, состоящий из абзацев и списков. При множественном использовании превращается в элемент ненумерованного списка. |

## 3. Хранение препромптов

### 3.1. Структура каталога препромптов

Препромпты организованы в иерархию:

```
.preprompts/               # корневая папка для всех препромптов
├── roles/                 # препромпты типа "роли"
│   ├── cpp-dev.md         # нейтральный по языку препромпт
│   ├── .ru/               # локализованные версии (русский)
│   │   └── cpp-dev.md
│   └── .en/               # локализованные версии (английский)
│       └── cpp-dev.md
├── objectives/            # препромпты типа "цели"
├── contexts/              # препромпты типа "контекст"
├── formats/               # препромпты типа "формат вывода"
├── extras/                # дополнительные препромпты
└── constraints/           # препромпты типа "ограничения"
```

Правила:
- Внутри каталога типа (например, `roles`) могут находиться только файлы `.md` и подкаталоги, имена которых начинаются с точки и состоят из двух букв (ISO 639-1, например `.ru`, `.en`). Любые другие подкаталоги игнорируются.
- Поддержки вложенных иерархий нет.

### 3.2. Расширения и frontmatter

- Файлы препромптов имеют расширение `.md`, кодировка UTF-8 (с корректной обработкой BOM).
- Файл может содержать YAML frontmatter, ограниченный строками `---` в начале. Всё, что после закрывающего `---` (или до конца файла, если закрывающий отсутствует), считается телом препромпта.
- Если frontmatter отсутствует, весь файл считается телом.

**Поля frontmatter:**
- `title` – название препромпта (используется в выводе `--list`).
- `description` – краткое описание.
- `author` – автор(ы) (строка или список).
- `date` – дата в формате YYYY-MM-DD.
- `extended` – `false`/`true`/`auto` (по умолчанию `auto`). Определяет, является ли препромпт расширенным.
- `requires` – зависимости (см. раздел 5).

### 3.3. Поиск корневого каталога проекта

Утилита определяет корень проекта следующим образом (начиная с текущего рабочего каталога и поднимаясь вверх):
1. Проверяется наличие файла `almai.yaml`. Если найден – это корень проекта.
2. Если `almai.yaml` отсутствует, проверяется наличие любого из каталогов-маркеров: `.git`, `.out`, `.vscode`, `.build`, `build`. Если такой каталог присутствует, поиск останавливается, и проект считается найденным (с пустым проектным файлом).
3. Если ничего не найдено, переход в родительский каталог. При достижении корня ФС проект не считается найденным.

### 3.4. Каталоги с препромптами проекта

Если существует каталог `$(ProjectRoot)/.preprompts.almai`, он используется как проектный каталог препромптов (имеет наивысший приоритет при поиске).  
Если такого каталога нет, но есть `$(ProjectRoot)/.preprompts`, используется он (с предупреждением о возможном конфликте с другими системами).

### 3.5. Каталоги дистрибутива и пользователя

Дистрибутив имеет структуру:

```
almai/
├── bin/                         # исполняемые файлы/скрипты
├── conf/                        # конфигурационные файлы
│   ├── almai.options            # основной конфиг (поставляется)
│   └── almai.custom.options     # системный конфиг (опционально)
├── preprompts/                   # базовые препромпты (поставляются)
├── preprompts.custom/            # системные препромпты (опционально)
└── ...
```

Пользовательские настройки и препромпты хранятся в `$HOME/.almai/`:
- `$HOME/.almai/.options` – пользовательский конфиг.
- `$HOME/.almai/preprompts/` – пользовательские препромпты.

## 4. Языковые настройки (локализация)

### 4.1. Определение текущего языка

Язык задаётся в следующем порядке приоритета (от высшего к низшему):
1. Опция командной строки `--lang=ru` (или `-l ru`).
2. Поле `lang` в файле проекта `almai.yaml`.
3. Опция `--default-lang=xx` (если указана).
4. Встроенное значение по умолчанию – `en`.

### 4.2. Поиск локализованного препромпта

При поиске препромпта с ID `cpp-dev` для языка `ru` проверяются пути (в порядке приоритета каталогов):
- `preprompts/roles/ru/cpp-dev.md`
- `preprompts/roles/cpp-dev.md`

Если локализованная версия найдена в более приоритетном пути, используется она, даже если в менее приоритетном есть нейтральная версия.

### 4.3. Локализация заголовков и вводных текстов

Для локализации используются идентификаторы строк. Заголовки разделов формируются динамически:
- `ТИП_ПРЕПРОМПТА-title` (например, `roles-title`) – заголовок раздела.
- `ТИП_ПРЕПРОМПТА-intro-simple` – вводный текст перед простыми препромптами.
- `ТИП_ПРЕПРОМПТА-intro-extended` – вводный текст перед расширенными препромптами.
- `project-descriptions-section-title` – заголовок секции с описанием проекта.

Переводы задаются опцией `--translation=ЯЗЫК:КЛЮЧ:ТЕКСТ` (например, `--translation=ru:roles-title:Твоя роль`). Если перевод не найден, используется английское название по умолчанию (первая буква заглавная, остальные строчные).

## 5. Зависимости препромптов

### 5.1. Поле `requires` в frontmatter

Поле `requires` может быть:
- строкой с логическим выражением;
- YAML-списком;
- отображением (dict) с ключами – типами препромптов и значениями – выражениями.

**Логическое выражение:**
- Запятая `,` – логическое И (AND).
- Вертикальная черта `|` – логическое ИЛИ (OR) внутри одного типа.
- Пример: `"cpp-dev|c-dev,java-dev"` означает `(cpp-dev OR c-dev) AND java-dev`.

**Форматы:**
```yaml
requires:
  roles: 'cpp-dev|c-dev,java-dev'
  constraints: 'cpp11'
```
или
```yaml
requires:
  roles:
    - cpp-dev|c-dev
    - java-dev
  constraints: cpp11
```
или (если все зависимости одного типа)
```yaml
requires: 'cpp-dev|c-dev,java-dev'
```

### 5.2. Алгоритм разрешения зависимостей

1. При добавлении препромпта проверяется его поле `requires`.
2. Если зависимость не удовлетворена (отсутствует в уже выбранных препромптах), она добавляется автоматически.
3. Для OR-зависимости выбирается первая существующая альтернатива (по порядку в выражении).
4. Если зависимость имеет тот же тип, что и текущий препромпт, она добавляется непосредственно перед ним (без изменения порядка других типов).
5. Проверка циклов: если при рекурсивном добавлении встречается уже обрабатываемый препромпт, выдаётся фатальная ошибка.
6. Если зависимость не найдена (нет файла), в обычном режиме выдаётся предупреждение и она игнорируется; в строгом режиме – ошибка и останов.
7. Если зависимость имеет неизвестный тип – аналогично п.6.

## 6. Типы препромптов и динамические опции

### 6.1. Список типов

Список типов препромптов задаётся опцией `--preprompt-types` (и аналогично в `almai.yaml`). Эта опция определяет:
- порядок разделов в итоговом документе;
- какие типы разрешены (опции для других типов игнорируются);
- какие динамические опции CLI добавляются.

Формат: `--preprompt-types=roles,objectives,contexts,...` (накопительно). Символ `-` очищает текущий список.

### 6.2. Динамические опции

Для каждого типа `T` (во множественном числе) создаются:
- `--T` / `--t` (где `t` – единственное число, полученное по правилам преобразования) – добавление препромптов по ID (можно несколько раз или через запятую).
- `--t-text` – добавление препромпта прямым текстом (однострочным).

**Преобразование единственного/множественного числа**:
- Используются словари, заполняемые опцией `--dict=ед:мн` (например, `--dict=analysis:analyses`).
- Если слова нет в словаре:
  - из единственного во множественное: добавить `s` (кроме случаев, когда слово уже заканчивается на `s` – считается множественным);
  - из множественного в единственное: удалить конечную `s`.
- Эти преобразования используются только для формирования имён опций; внутреннее представление всегда множественное число.

### 6.3. Универсальная опция `-p` / `--preprompt`

Позволяет добавлять препромпт без указания типа. Алгоритм:
1. Для каждого типа из текущего списка `preprompt-types` проверяется наличие файла `<ID>.md` в соответствующем каталоге (с учётом языка).
2. Если файл найден ровно в одном типе – добавляется как обычный препромпт этого типа.
3. Если найден в нескольких типах – ошибка (выводятся конфликтующие типы), программа завершается.
4. Если не найден ни в одном – предупреждение (или ошибка в строгом режиме).

## 7. Сборка итогового документа

### 7.1. Порядок разделов

Разделы следуют в порядке, заданном `--preprompt-types`. Для каждого типа:
- Если есть хотя бы один препромпт (или текстовый), создаётся раздел.
- Заголовок раздела берётся из локализации (`тип-title`) или формируется по умолчанию.
- Сначала выводятся простые препромпты, затем расширенные (с соответствующими вводными текстами, если заданы).

### 7.2. Вставка простых препромптов

- Если препромпт один – вставляется как есть.
- Если несколько – каждый превращается в элемент ненумерованного списка:
  - Удаляются ведущие и завершающие пустые строки.
  - Первая строка получает префикс `- `.
  - Последующие строки получают отступ в два пробела (таким образом внутренние списки становятся вложенными).
  - Серии из нескольких пустых строк схлопываются в одну.

**Ограничение:** простой препромпт не может состоять только из списка без вводного текста – это считается ошибкой.

### 7.3. Вставка расширенных препромптов

- Удаляются ведущие и завершающие пустые строки.
- Уровень всех заголовков ATX повышается на один (добавляется `#` в начало). Заголовки уровня 6 (`######`) превращаются в жирный текст (`**текст**`).
- Поддерживаются только ATX-заголовки; Setext не поддерживаются.
- Расширенные препромпты вставляются последовательно, каждый со своим содержимым, без дополнительных списков.

### 7.4. Вставка описаний проекта

Файлы описаний (например, `PROJECT.md`, `REQUIREMENTS.md`) задаются опцией `--project-descriptions` (список имён без расширения). Они ищутся:
- в каталоге проекта (где лежит `almai.yaml`);
- при наличии каталога `.project-descriptions.almai` или `.project-descriptions` – в нём.

Каждый файл вставляется по правилам расширенных препромптов, но без вводного текста. Пустые файлы игнорируются.

### 7.5. Вывод

По умолчанию результат выводится в `STDOUT`. Опция `--output FILE` (`-o`) перенаправляет вывод в файл.

## 8. Интерфейс командной строки

### 8.1. Основные опции

| Опция | Описание |
|-------|----------|
| `-h, --help` | Справка. |
| `--lang LANG, -l LANG` | Язык промптов (например, `ru`, `en`). |
| `--default-lang LANG` | Язык по умолчанию, если не задан иначе. |
| `--preprompt-types LIST` | Список типов препромптов (через запятую). Можно указывать несколько раз. `-` очищает список. |
| `--dict SINGLE:PLURAL` | Добавляет пару единственное→множественное число для формирования опций. |
| `--translation LANG:KEY:TEXT` | Добавляет локализованный текст для ключа. |
| `--preprompts-dir DIR` | Добавляет каталог с препромптами в путь поиска (можно несколько раз). |
| `--project-descriptions LIST` | Список имён файлов описаний проекта (без `.md`). `-` очищает список. |
| `--no-project-preprompts` | Игнорировать препромпты, заданные в `almai.yaml`. |
| `--output FILE, -o FILE` | Записать результат в файл вместо `STDOUT`. |
| `--list[=types]` | Вывести список доступных препромптов. Без аргумента – все типы. |
| `--verbose-list` | При `--list` показывать полные пути к файлам. |
| `--strict[=MODE]` | Включить строгий режим (останов при любой ошибке). MODE: `true/false/1/0/on/off/yes/no`. |
| `--no-strict` | Отключить строгий режим. |
| `--tokenizer-lib LIB` | Использовать конкретную библиотеку для подсчёта токенов (`tokenizers`, `rs-bpe`, `smolbpe`). |
| `--tokenizer-lib-list` | Показать доступные библиотеки токенизации. |
| `--no-builtin-options` | Игнорировать основной конфиг `almai.options`. |
| `--no-custom-builtin-options` | Игнорировать системный конфиг `almai.custom.options`. |
| `--no-user-builtin-options` | Игнорировать пользовательский конфиг `$HOME/.almai/.options`. |

### 8.2. Динамические опции для типов

Для каждого типа `T` (мн.ч.) генерируются:
- `--T ID[,ID...]` или `--t ID` – добавить препромпты по ID.
- `--t-text TEXT` – добавить текстовый препромпт.

Пример:
```
--roles=cpp-dev,devops --objective analyze --constraint-text="Используй C++17"
```

### 8.3. Файлы опций

Аргумент, начинающийся с `@`, интерпретируется как путь к файлу опций. Файл содержит по одной опции на строку (как в командной строке), поддерживаются комментарии (`#` в начале строки) и пустые строки. Возможно вложенное подключение (через `@`), циклы отслеживаются и вызывают ошибку.

### 8.4. Макросы в конфигурационных файлах

В конфигах поддерживаются макросы вида `$(ИМЯ)`. Раскрываются только при обработке файлов опций. Доступные имена:
- `AppName` – имя приложения (`almai`).
- `AppRoot`, `AppPath`, `AppRootPath` – путь к корню дистрибутива.
- `AppBinPath`, `AppBin` – путь к каталогу с исполняемыми файлами.
- `AppConfPath`, `AppConf` – путь к конфигурационным файлам дистрибутива.
- `UserHome` – домашний каталог пользователя.
- `UserAppPath` – `$HOME/.almai/`.
- `ProjectRoot` – корень проекта (если определён; иначе пустая строка).
- `CWD` – текущий рабочий каталог.

Неизвестный макрос – фатальная ошибка.

## 9. Проектный файл `almai.yaml`

Файл YAML в корне проекта может содержать:

```yaml
lang: ru                                    # язык проекта
preprompt-types: [roles, objectives]        # типы препромптов (аналогично CLI)
project-preprompts:                          # препромпты по умолчанию
  roles: [cpp-dev, devops]
  objectives: analyze
project-descriptions: ["PROJECT", "REQUIREMENTS"]  # файлы описаний
```

- `project-preprompts` – добавляются до опций CLI; при дублировании оставляется первое вхождение.
- `project-descriptions` – аналогично CLI, `-` очищает список.

Проектный файл обрабатывается после встроенных конфигов, но до командной строки.

## 10. Обработка ошибок

### 10.1. Режимы

- **Обычный режим** (`--strict=false`): при ошибках (кроме циклических зависимостей) выдаётся предупреждение в stderr, обработка продолжается.
- **Строгий режим** (`--strict`): любая ошибка приводит к останову и коду возврата 1.

### 10.2. Критичные ошибки (всегда фатальны)

- Циклические зависимости.
- Отсутствие каких-либо препромптов и файлов описаний (нечего генерировать).
- Синтаксическая ошибка в frontmatter (в строгом режиме; в обычном – предупреждение и файл обрабатывается как обычный текст).

### 10.3. Коды возврата

- `0` – успех.
- `1` – ошибка (в т.ч. фатальная).
- (Опционально) можно расширить: `2` – ошибка конфигурации, `3` – ошибка зависимостей и т.д.

## 11. Подсчёт токенов

### 11.1. Эвристический алгоритм

Если нет библиотек для точного подсчёта, используется встроенная эвристика:
- Последовательность букв (Unicode) – 1 токен.
- Последовательность цифр – 1 токен.
- Если цифры и буквы чередуются, разделяются: цифры → токен, буквы → токен.
- Дефис (кроме ASCII минуса) между буквами – часть токена; ASCII минус – разделитель, кроме случаев с нелатинскими буквами (например, «кто-нибудь» – один токен).
- Апостроф между латинскими буквами – часть токена.
- Последовательности одинаковых знаков пунктуации (например, `---`) – один токен.
- Стандартные операторы (`--`, `++`, `==`, `!=`, `<=`, `>=`, `<=>`, `->`, `=>`, `::`) – один токен.
- Остальные одиночные символы – отдельные токены.

Результат выводится в stderr: `[ heuristic token count: N ]`.

### 11.2. Подсчёт с помощью библиотек

При наличии библиотек `tokenizers`, `rs-bpe` или `smolbpe` выбирается первая доступная (или указанная через `--tokenizer-lib`). Для неё используются параметры, приближенные к GPT-2/4 (например, модель `gpt2` для `tokenizers`). Результат выводится в stderr: `[ <lib> token count: N ]`. Если библиотека не может быть использована, выдаётся предупреждение и применяется эвристика.

## 12. Технические требования к реализации

### 12.1. Версия Python

- Минимальная версия: Python 3.7.
- Желательно обеспечить максимальную совместимость в пределах 3.7–3.11+.

### 12.2. Сторонние библиотеки

- Разрешено использовать только стандартную библиотеку, кроме:
  - `pyyaml` – для обработки YAML (обязательно). Если не установлена – выводить инструкцию по установке.
  - Опционально: библиотеки для подсчёта токенов (не обязательны).
  - Для тестов: `pytest` (не входит в дистрибутив).

### 12.3. Архитектура (рекомендуемая)

Желательно разделить на модули:
- `config_manager` – загрузка конфигураций (файлы, CLI, макросы).
- `preprompt_loader` – поиск и чтение файлов с учётом языка и приоритетов.
- `dependency_resolver` – разрешение зависимостей, проверка циклов.
- `prompt_assembler` – сборка итогового Markdown.
- `token_counter` – подсчёт токенов (эвристика + внешние библиотеки).
- `cli` – обработка аргументов (можно использовать `argparse` или `click`).
- Основной скрипт (`almai.py`) координирует работу модулей.

### 12.4. Кодировки

- Все входные файлы должны быть в UTF-8 (с корректной обработкой BOM). Обнаружение другого BOM – ошибка (файл игнорируется).
- Вывод – UTF-8 без BOM.

### 12.5. Платформенная совместимость

- Для определения домашнего каталога использовать `pathlib.Path.home()`.
- Работа с путями через `pathlib` (кроссплатформенно).
- В конфигурационных файлах допускаются как прямые, так и обратные слеши (под Windows).
