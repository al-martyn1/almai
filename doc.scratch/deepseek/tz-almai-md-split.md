# Твоя роль
Ты:
- опытный программист на языке Python;
- отличный архитектор программного обеспечения.


# Цель
Создать скрипт на языке Python, который
на входе markdown-документ (".md"), содержащий листинги, непосредственно перед каждым из которых 
вставлено имя файла (включая подкаталоги),
и сохраняет эти листинги в отдельные файлы согласно их именам, создавая по необходимости
подкаталоги.

Листинги могут обозначаться:
- тройным бэктиком
- тройной тильдой

Листинги с отступом в 4 пробела или табуляцией - не обрабатываются (не сохраняются), следует выдавать предупреждение при каждом нахождении подобного листинга.
Достаточно выдавать предупреждение каждый раз, когда встречается строка, начинающаяся с 4 пробелов или табуляции, и при этом мы не находимся внутри блока, 
ограниченного маркерами листинга.

Указание языка листинга в открывающем маркере допустимо, но никак не влияет на обработку.

Следует поддерживать экранирование маркеров листинга, как в markdown (при это допустимо экранирование только первого символа маркера).



Имя файла может быть обрамлено кавычками (или другим оформлением, например двойным '*' - выделение жирным).
Имя файла может начинаться с пробелов или табуляций, в этом случае ведущие пробельные символы удаляются.
В строке с именем файла не должно быть больше никакой информации (кроме возможных пробельных символов).
В строке с именем файла может быть указано только одно имя файла. Нельзя один листинг сохранить в несколько файлов.
Между именем файла и листингом с его содержимым не может быть никаких строк, включая пустые.
Пустыми строками считаются как строки, ничего не содержащие, так и строки, содержащие только пробелы или/или табуляции.
Если перед листингом расположена пустая строка, считается, что этот листинг не отмечен именем файла.

Допустимое обрамление:
- двойные кавычки ("имя")
- одинарные кавычки ('имя')
- двойные звёздочки (жирный, **имя**)
- одинарные звёздочки (курсив, *имя*)
- обратные кавычки (код, `имя`)

В строке с именем файла могут быть завершающие пробелы и/или табуляции. Их следует удалить до всей остальной обработки.

Имя файла считается корректно обрамлённым, если маркер обрамления в начале соответствует маркеру обрамления в конце.

Если после удаления маркеров обрамления остаётся пустая строка имени файла, имя файла следует сгенерировать автоматически, выдав предупреждение.

Имя файла может быть указано без обрамления. В этом случае оно должно содержать хотя бы одну точку или разделитель пути,
и не должно содержать пробелов и недопустимых символов. Если в имени файла встречаются пробелы или недопустимые 
символы, это считается описанием демонстрационного листинга, а не файла для сохранения. В этом случае мы всё равно пытаемся
сохранить этот листинг в файл. Имя файла в этом случае пытаемся создать из описания файла. Следует проверить, есть ли
расширение, если его нет, используем язык листинга или .txt, как при автоименовании. Производим "нормализацию" сформированного имени.

Если строка, предположительно содержащая имя файла (без обрамления), содержит пробелы или недопустимые символы, 
она не считается корректным именем, а интерпретируется как описание. Для генерации имени файла используется эта 
строка целиком после применения к ней процедуры нормализации полностью (описанна ниже), включая замену пробелов и недопустимых 
символов на _. Затем проверяется наличие расширения в полученном имени (точка с последующими символами после последнего 
разделителя пути). Если расширение отсутствует, оно добавляется на основе языка листинга (если указан) или .txt 
(по аналогии с автоименованием). Полученное имя далее участвует в проверке на уникальность и разрешении коллизий.

Примерный алгоритм обработки имени/строки без обрамления:
- Если строка содержит пробелы или недопустимые символы - строка является описанием, генерируем имя.
- Иначе (нет пробелов и недопустимых символов)  используем как имя файла и проводим нормализацию.

Пробелы в имени файла допустимы при использовании "обрамления".

Если маркер обрамления в начале имени файла не соответствует маркеру в конце (или в конце маркер отсутствует), то следует удить маркер(ы),
как и в случае отсутствия ошибок, и выдать предупреждение.

Все пути являются относительными относительно выходного корневого каталога, задаваемого опцией --output-dir.

Если опция --output-dir не указана, то выходным каталогом считается тот каталог, в котором расположен входной файл.

Если имя входного файла не содержит абсолютный путь, оно считается относительным относительно текущего рабочего каталога.

Если блок кода (листинг) не предварён именем файла, следует выдать предупреджение в STDERR, и автоматически сгенерировать имя на основании
глобального счётчика (начальное значение 0). Для расширения имени файла в этом случае следует использовать язык блока кода, если он указан, или ".txt".
Формат имени файла: 'autosave_XX.EXT', где XX - две десятичных разряда значения счётчика (для больших значений счётчика выводится больше разрядов). 
Счётчик для автоматически именуемых файлов является глобальным для всех языков.
Такое имя после генерации также должно пройти процедуру "нормализации".
При совпадении сгенерированного имени с уже использованным (в рамках текущего документа, включая те, что заданы явно) счётчик увеличивается 
до тех пор, пока имя не станет уникальным.

Имя языка преобразовывается в расширение по словарю. Словарь должен лежать в каталоге "../conf" относительно исполняемого файла скрипта под именем 'almai-md-split.dict'.
Пример расположения файлов:
```
/usr/local/bin/almai-md-split.py
/usr/local/conf/almai-md-split.dict
```

или 
```
/usr/local/share/almai-md-split/bin/almai-md-split.py
/usr/local/share/almai-md-split/conf/almai-md-split.dict
```

Но лучший способ — использовать библиотеку platformdirs (или её предшественника appdirs) для определения правильных путей для конфигурационных файлов в зависимости от ОС.
При этом следует обрабатывать два файла словарей - системный и пользовательский (site_data_dir и user_config_dir).

Если файл системный или пользовательский файл almai-md-split.dict не находится, следует вывести предупреждение, 
что он не найден, с полным именм файла, где скрипт пытается его прочитать.
В режиме вывода справки (--help) также следует выводить имя данноых файлов: "Dictionary file: ...".

Следует предусмотреть опцию --dict=ИМЯ_ФАЙЛА, которая позволяет добавить файл ИМЯ_ФАЙЛА как словарь расширений. Этот словарь загружается дополнительно к системному и пользовательскому.

Это простой текстовый файл. Пустые строки (ничего не содержащие, или состоящие только из пробелов и табуляций) - игнорируются.
Строки, начинающиеся первым значимым символом (первым символом после удаления ведущих пробелов ) '#' - комментарии, также игнорируются. 
Остальные строки содержат разделяемую пробелом пару: РАСШИРЕНИЕ НАЗВАНИЕ ЯЗЫКА.
Пример:
```
.cpp C++
py   Python
.py2   Python 2
.py3   Python 3
```
Расширение может иметь ведущую точку, или быть без него. Первый токен в строке, до пробела - расширение. Названием языка считается остаток строки. 
Ведущие и завершающие пробелы в названии языка удаляются.
В итоге у нас должен быть словарь: НАЗВАНИЕ ЯЗЫКА -> РАСШИРЕНИЕ. Ведущая точка удаляется для унификации перед помещением в словарь. В дальнейшем
точка всегда добавляется при добавлении расширения к имени файла.

НАЗВАНИЕ ЯЗЫКА - регистронезависимое. При добавлении элементов в словарь НАЗВАНИЕ ЯЗЫКА следует приводить к нижнему регистру, при поиске в словаре - тоже.

Если название языка отсутствует в словаре, то имя языка используется как расширение, и выдаётся предупреждение.

Если файл словаря отсутствует или поврежден - выдаётся предупреждение. Если словарь содержит некорректные строки, они игнорируются, и выводится предупреждение.

Если имя файла содержит элементы, которые помещают файл за пределы выходного каталога, а также какие-то недопустимые символы, 
то такие элементы следует удалить. При этом в STDERR должно быть выдано предупреждение.
Алгоритм нормализации:
- В качестве разделителя пути может использоваться как прямой так и обратный слэш. Следует все разделители пути заменить на прямой слэш ('/').
- Если указан абсолютный путь ("/usr/home/user/project", "~/project"), то путь следует сделать относительным:
  "usr/home/user/project", "project"). Если указан диск (путь в стиле Windows) - удалить часть, содержащую букву диска и двоеточие, а также первый разделитель пути, 
  следующий непосредственно за обозначением диска, если он есть.
- Символ '~' не надо разрешать в домашний каталог пользователя. 
- Сетевые абсолютные пути также следует преобразовать в относительные, удаляя все ведущие разделители, но сохраняя структуру: "\\server\share\file.cpp" -> "server\share\file.cpp".
- Если в пути используются элементы '..', то следует обрабатывать как подъём на уровень вверх. Если при подъеме на уровень вверх происходит выход за пределы
  выходного каталога, то подъём игнорируется, так, имя "a/../../b.cpp" должно стать "b.cpp".
  При разборе имени у нас есть список частей пути, в который мы добавляем новую часть, или удаляем последнюю, когда встречается "..". 
  Если список пуст, то удаления не происходит.
- Недопустимые символы в именах файлов заменяются на символ '_' (недопустимые символы - \ / : * ? " < > | (Windows) + управляющие символы).
  Поведение утилиты должно быть одинаково на всех системах, поэтому ориентируемся на более строгие ограничения Windows.
  Замена недопустимых символов на '_' производится в каждом компоненте пути отдельно, разделители пути сохраняются.
- Пробелы заменяются на символ '_', если при задании имени файла не использовалось "обрамление".
- Ведущие и завершающие точки и пробелы удаляются в каждом компоненте пути.
- Если система не поддерживает в именах файлов символы, отличные от латиницы, то следует произвести транслитерацию.
  Если нет возможности это определить или это сложно и ненадёжно, то считаем, что система достаточно современна, и поддерживает национальные символы в именах файлов.
  Если нет библиотеки транслитерации, то все проблемные символы заменяются на символ '_'.

Нормализация производится для всех имен файлов - с обрамлением и без, но существование обрамления должно учитываться при нормализации.
Если после нормализации получается пустое имя файла, то следует автоматически сгенерировать имя файла (по описанным выше правилам автоматического именования файлов),
и выдать предупреждение, выведя в нём оригинальное имя файла и сгенерированное.

Добавление имени корневого каталога ко всем путям производится только непосредственно перед записью файла.

В Windows существуют зарезервированные имена устройств (CON, PRN, AUX, NUL, COM1..9, LPT1..9), которые нельзя использовать для имён файлов (даже с расширением, например CON.txt).
Зарезервированные имена являются зарезервированными независимо от регистра.
Такие файлы следует переименовывать, добавляя префикс "renamed_" к нимени файла, например: "md/LPT1.md" -> "md/renamed_LPT1.md" и тп. 
Данное ограничение относится ко всем компонентам пути (как к подкаталогам, так и имени файла)

Сначала применяется нормализация, затем проверка на зарезервированные имена (и переименование), потом разрешение коллизий (добавление счётчика).

Для работы с путями предпочтительно использовать pathlib.

Разрешение символа '~' в домашний каталог пользователя не допускается при задании имен файлов в md-документе, но может
использоваться при вызове утилиты в опциях командной строки, например, при вызове с опцией '--output-dir=~/tmp1') - этот символ может разрешать оболочка.
В Windows такое поведение не поддерживается, но может работать в bash для Windows. При обработке опций мы полагаемся на вызывающую оболочку.

Если в документе два или более файлов имеют одно и то же имя (после приведения к допустимому виду, нормализации), 
то следует завести внутренний счётчик (с именем файла, включая путь и расширение), и значение
счетчика (два десятичных разряда) вставлять при сохранении перед расширением файла через подчёркивание, кроме самого первого файла, 
например: "file.cpp", "file_01.cpp", "file_02.cpp". Расширение определяется как часть после последней точки с учётов разделителей пути 
(точка должна быть после последнего разделителя пути). Если расширение не найдено, счётчик добавляется в конец имени.
При совпадении сгенерированного имени с уже использованным счётчик увеличивается до тех пор, пока имя не станет уникальным.

При проверке на повторения имён файлов следует игнорировать регистр символов (Windows-поведение) - поведение утилиты должно быть одинаково на всех системах.

Подкаталоги создаются автоматически.

Кодировка предполагается UTF-8. Допустимо наличие соответствующей BOM, которая удаляется перед дальнейшей обработкой.

Если листинг пустой, создаётся пустой файл. Это может использоваться для создания структуры каталогов, используя пустые файлы с именем, например, dummy.txt в разных каталогах.

Допустимы вложенные листинги (например, у нас на листинге - маркдаун-документ, который, в свою очередь, содержит листинги).
Следует:
- отслеживать текущий уровень вложенности и количество символов открывающего маркера.
- закрывающий маркер должен содержать то же количество символов, что и открывающий для данного уровня, и использовать тот же маркерный символ (маркеры должны быть идентичны).
- внутри блока кода любой текст (в том числе маркеры с другим количеством символов) считается содержимым.

Если блок кода не заканчивается до конца файла, следует считать конец файла такде окончанием листинга, и выдать предупреждение.


Если в процессе работы возникают какие-либо ошибки (например, ошибки ввода-вывода) - следует выдать соответствующее предупреждение.

Если в процессе работы происходят фатальные ошибки, программа завершается с ненулевым кодом возврата. Уже созданные и сохранённые файлы 
остаются.

Если входной файл не задан, следует выдать сообщение об ошибке и заверершить работу.

Если входной файл не не в кодировке UTF-8 или произошли ошибки при декодировании, следует выдать сообщение об ошибке и заверершить работу.

Пример вызова утилиты (задаём выходной каталог):
```
almai-md-split document.md --output-dir=~/tmp1
```

или так:
```
almai-md-split --output-dir=~/tmp1 document.md
```

Пример вызова утилиты (используется каталог, в котором расположен файл):
```
almai-md-split ~/tmp1/document.md
```

За один запуск обрабатывается только один файл.

HTML-комментарии (фрагменты, заключённые в <!-- -->) должны игнорироваться.

# Технические требования

##  Общая архитектура и модули

Важно, чтобы программа состояла из определенных логических блоков для облегчения ревью и доработки, 
но детали реализации внутри этих блоков не задаются жестко и оставляются на усмтрение разработчика.

Желательно (но не обязательно) использовать ООП и классы, и разные модули разнести по разным файлам, которые включаются в основной файл скрипта.

## Версия Python
Версия Python должна быть не ниже 3.7, но не хотелось бы фиксироваться на какой-то конкретной подверсии, надо сделать максимально совместимо.

## Разбор опций командной строки

Следует использовать argparse. Между именем опции и значением должен стоять знак равно ('='), использование пробела нежелательно.
Но если отключить использование пробела сложно, то его использование допустимо.

## Дополнительные библиотеки

По возможности следует использовать только встроенные библиотеки. Допустимо, при необходимости, использовать библиотеки platformdirs и unidecode.
Если библиотека platformdirs не установлена, следует выдать предупреждение, а также инструкцию по установке библиотеки.
В случае отсутствия platformdirs словарь ищем в одном месте, по пути '../conf' относительно месторасположения исполняемого скрипта программы.

## Платформенная совместимость

Для работы с путями предпочтительно использовать pathlib.

## Целевая ОС
Результирующий скрипт должен работать в операционных системах Windows и Linux, а также в Unix-подобных (например, FreeBSD).

## Код возврата

При наличии ошибок должен возвращаться ненулевой код возврата (1). Предупреждения не влияют на код возврата.

## Ошибки и предупреждения

При выводе ошибок и предупреждений их следует предварять префиксами "error: " и "warning" соответственно.

Фатальными являются ошибки, которые не позволяют продолжить работу, например, не найден или не читается входной файл.

# Тесты

Для тестирования следует использовать библиотеку pytest.
Если библиотека pytest не установлена, то следует выдать сообщение с информацией по её установке.

Следует написать следующие тесты:
- Модульные тесты (unit tests) – проверка отдельных компонентов в изоляции.
- Интеграционные тесты – проверка взаимодействия компонентов и работы с реальной файловой системой.
- Тесты командной строки (CLI) – проверка обработки аргументов, опций, генерации вывода.
