# Твоя роль
Ты:
- опытный программист на языке Python;
- отличный архитектор программного обеспечения.


# Цель
Создать скрипт на языке Python, который
по заданным ID собирает препромпты для ИИ в единый markdown документ.

Название проекта - 'almai'.

ID могут быть следующих типов: 
- roles - твои роли
- objectives - цели проекта
- contexts - контексты задачь
- formats - выходные форматы
- extras - дополнительные уточнения
- constraints - ограничения

Например, роли (roles) могут быть такими:
- cpp-dev - разработчик на C++ (препромпт: "опытный разработчик на языке C++");
- py-dev - разработчик на Python (препромпт: "опытный разработчик на языке Python");
- software-architect (sw-architect) - архитектор программного обеспечения (препромпт: "опытный архитектор программного обеспечения").
- devops - специалист по автоматизации (препромпт: "devops - специалист по автоматизации сборки, тестирования и развёртывания (обновления) программ");
- cmake - знаток CMake (препромпт: "ты эксперт по CMake, знаешь все тонкости написания CMakeLists.txt").

Ограничения (constraints), например, могут быть такими:
- cpp17 (препромпт: "используй C++ 17");
- py2 (препромпт: "используй Python версии 2").

Цели (objectives), например, могут быть такими:
- analyze (препромпт: "проанализируй архитектуру");
- err-review (препромпт: "проверь код на наличие ошибок");
- perf-review (препромпт: "проверь код на предмет возможности увеличения производительности, найди все узкие по производительности места");

Пре-промты лежат в нескольких локациях:
- подкаталог .preprompts в месте расположения программы;
- подкаталог .preprompts.custom в месте расположения программы;
- подкаталог .preprompts в месте расположения рабочего проекта;
- задаваемые пользователем каталоги.

Каталоги проверяются в порядке, обратном порядку добавления в список поиска, для того, чтобы пользователь мог
переопределить предварительно настроенные препромпты.

Структура каталога .preprompts выглядит так:

```
.preprompts/               # корневая папка для всех препромптов
├── roles/                 # директивы #!ai role ...
│   ├── cpp-dev.md         # Language-neitral препромпт для роли "C++ программист"
│   ├── cpp-programmer.md
│   ├── software-architect.md
│   ├── .ru                # каталог для локализованных препромптов
│   │   └── cpp-dev.md     # Локализованный (русский) препромпт для роли "C++ программист"
│   └── .en                # каталог для локализованных препромптов
│   │   └── cpp-dev.md     # Локализованный (английский) препромпт для роли "C++ программист"
├── objectives/            # #!ai objective ...
│   ├── analyze.md
│   ├── review.md
│   ├── generate-tests.md
│   ├── .ru                # каталог для локализованных препромптов
│   └── .en                # каталог для локализованных препромптов
│   └── ...
├── formats/               # #!ai format ...
│   ├── code.md
│   ├── detailed-write-up.md
│   ├── markdown-table.md
│   ├── .ru                # каталог для локализованных препромптов
│   └── .en                # каталог для локализованных препромптов
│   └── ...
├── extras/                # #!ai extra ...
│   └── (можно оставить для параметризованных дополнений)
│
└── (другие категории, например, contexts)
```

В каталогах '.preprompts/NNN' (NNN - какой-то отдельный тип препромптов) могут быть подкаталоги '.ru', '.en' и тп, а в них - 
локализованные препромпты данного типа. Эти каталоги отвечают за поддержку языка генерируемого промпта, и поиск сначала 
производится в них, а потом уже в корне каталога '.preprompts'. Локализаций (языковых подкаталогов) может быть произвольное
количество, по числу необходимых локализаций.
Локали в полном формате (en-US, en-GB) - не поддерживаются, используем только primary язык (двухбуквенный ISO-код).

Каталоги с локализациями препромптов должны начинаться с точки, после которой идёт двухбуквенный ISO-код языка.

Внутри каталога типа (например, roles) могут находиться только:
- файлы с расширением .md (препромпты);
- подкаталоги, имена которых состоят из двух букв (коды языков по ISO 639-1), предвварённые точкой.
- любые другие подкаталоги игнорируются (не участвуют в поиске).

Никакие иерархии в каталогах препромптов не поддерживаются.


# Глоссарий

- Препромпт (preprompt) - представляет из себя некоторый текст, который должен быть
  добавлен при формировании запроса к ИИ до описания задачи.
- Тип препромпта (preprompt type) - тип препромпта описывает, к какой категории относится препромпт: задаёт роль ИИ,
  задаёт ограничения, задаёт дополнительные условия, задаёт цели, задаёт тип выходного документа (выходных документов), и т.п.
- Идентификатор препромпта (preprompt ID) - однозначно идентифицирует препромпт в рамках конкретного типа препромптов; 
  может содержать латинские буквы, цифры, знак минус и символ подчеркивания; используется для ссылки на конкретный препромпт
  (обычно для добавления текста данного препромпта в конечный запрос).
- Категория препромпта - синоним типа препромпта.


# Результат

препромпты должны быть скомбинированы по разделам (обозначеным заголовком первого уровня).
Например:

```md
# Твоя роль
Ты:
- опытный программист на языке C++;
- опытный архитектор программного обеспечения.


# Цель

Проанализировать архитектуру проекта.


# Результат

В результате работы требуется получить набор UML-диаграм на языке PlantUML.

...
```

Желательно, чтобы утилита комбинировала препромпты в связанный текст, но, если это сложно сделать 
локально средствами Python, то можно сделать просто перенос текста из библиотеки в документ.
Если препромптов несколько в каком-либо разделе, то их нужно организовать в ненумерованный список.
Если препромпт содержит список, то надо создать вложенный список.

Подробно вставка препромптов описана в разделе "Вставка препромптов в результирующий документ".

Результат должен выводится в `STDOUT`, или в файл, если задана соответствующая опция (`--output`/`-o`).


# Хранение препромптов в библиотеке препромптов

Предполагается, что каждый препромпт - это текст в c поддержкой разметки markdown, которая используется 
в параграфах (жирный/курсив/подчеркнутый шрифт), без структурных элементов типа заголовков, листингов, и тп.
Возможно, могут понадобится ненумеорованные списки.

Если требуется структурная организация препромпта (заголовки, листинги, и тп), следует использовать расширенные препромпты.

Имя файла препромпта (без расширения) является его идентификатором, по которому мы ссылаемся на препромпт. Например, у нас
есть файл 'roles/cpp-dev.md', тогда для задания роли мы используем идентификатор 'cpp-dev'.

Файл препромпта может содержать frontmatter в формате YAML, там могут быть уточнения по обработке препромпта, 
а также какая-то метаинформация.


## Коллизии имён препромптов

Могут существовать препромпты 'roles/analyze.md' и 'objectives/analyze.md'. Это не вызывает проблемы, так как мы будем их
подключать, используя динамически добавляемые опции: `--role=analyze --objectives=analyze`.


## Расширения файлов и frontmatter

- Файлы препромптов имеют расширение .md.
- Файлы препромптов могут содержать YAML frontmatter, ограниченный строками --- в начале файла. Всё, что после второго --- (или до конца файла, если закрывающий --- отсутствует), считается телом препромпта.
- Если frontmatter нет, весь файл считается телом.


## Содержимое frontmatter

Frontmatter может содержать следующие поля:
- title - название
- description - краткое описание. Пока не определено, когда оно будет использоваться. В текущий момент тут можно просто описать препромпт немного подробнее, чем в title.
- author - список авторов. Следует поддержать как одиночное значение в виде строки (если автор один), так и список.
- date - дата создания/обновления, в формате ISO YYYY-MM-DD
- extended - указывает, является ли препромпт расширенным или обычным.
  Может принимать значения false/true/auto. По умолчанию используется значение auto - автопределение типа препромпта.
- requires - зависимости.


Frontmatter не является обязательным и может отсутствовать. Если frontmatter присутствует, текстом препромпта является весь текст,
следующий за строчкой, содержащей закрывающий '---'.

### requires
Поле requires содержит список препромптов, которые должны быть включены перед данным препромптом.
Допустим, у нас есть препромпт 'constraints/cpp17.md'. Для его использования нам перед этим надо указать, что у ИИ есть роль cpp-dev, 
иначе constraint:cpp17 просто не имеет смысла. Нам нужно указать, cpp17 требует использования 'roles/cpp-dev.md'.

Зависимости могут (а скорее, всегда будут) "пересекать" типы препромптов: в примере выше ограничение 'constraints/cpp17.md' требует 
роли 'roles/cpp-dev.md', поэтому требуется словарь по типам препромптов.

Система препромптов должна быть согласованной. Это значит, что не должно быть ссылок на типы препромптов, которые не заданы
в текущей системе препромптов. В том числе, если в requires есть ссылка на неизвестный при текущих настройках тип препромпта,
то должна генерироваться фатальная ошибка.

### Пример frontmatter
```
requires:
  roles: 'cpp-dev|c-dev,java-dev'
  constraints: 'cpp11'
```

В примере приведён сложный пример задания зависимости: требуется, чтобы до данного препромпта был включён препромпт 
для роли (roles) cpp-dev или c-dev, **и** препромпт для java-dev. Также должен быть подключен constraints препромпт cpp11.

Запись 'cpp-dev|c-dev,java-dev', приведённая для типа препромптов roles в примере выше, является строкой, и должна обрабатываться 
как логическое выражение  '(cpp-dev || c-dev) && (java-dev)' - символ '|' - это ИЛИ (OR), символ ',' (запятая) - это И (AND).

OR-зависимости могут быть только внутри одного типа, пересечение типов при указании OR-зависимости не допускается
(нет способа внутри строки зависимости указать тип препрпромпта, он задаётся ключём YAML-отображения).

Алгоритм преобразования строки зависимости в логическое выражение: 
- разбиваем по запятым в список (и удаляем пробелы до и после);
- для каждого элемента списка производим преобразование:
  - заменяем вертикальную черту '|' на оператор OR (ИЛИ) - разбиваем на список, удаляем пробелы до и после, собираем обратно с оператором;
  - окружаем данный элемент скобками;
- собираем список обратно с использованием оператора AND (И).

### Пример frontmatter
```
requires:
  roles: 'c-dev,java-dev'
```

В данном примере нет альтернативы из предыдущего примера - C++ или C программист, в данном случае однозначно задаются роли: c-dev AND java-dev
(возможно, требуется библиотеку на C подключить к Java проекту).

Если зависимость не была подключена до использования текущего препромпта, то её надо подключить автоматически (выдав уведомление).
Если у нас список альтернатив (через '|') - следует подключить первую существующую альтернативу (по порядку следования в requires-выражении).

Если зависимость имеет другой тип, она просто добавляется в соответствующий список препромптов своего типа. 
Если зависимость имеет такой же тип, как и текущий препромпт, она должна быть добавлена непосредственно перед текущим препромптом.
Никакой топологической сортировки или частичного упорядочивания не требуется, требуется простая проверка циклов.
Если зависимость уже была добавлена, ничего делать не требуется.

Для обнаружения циклов достаточно простого обнаружения, что при рекурсивном добавлении мы снова наткнулись на исходный препромпт.

Если встречается несколько неподключенных зависимостей, то они подключаются по мере указания в requires.

Если зависимость уже подключена, ничего делать не требуется.

Если OR зависимость не может быть удовлетворена (отсутствуют все альтернативы), то выдаётся предупреждение или ошибка (в зависимости от --strict),
а зависимость игнорируется.

Более сложные выражения, в том числе группирущие скобки, поддерживать не требуется.

Если зависимость имеет неизвестный тип, выдаётся предупреждение или ошибка (в зависимости от --strict), а зависимость игнорируется.

Сначала производится сборка зависимостей по типам, используя в начале препромпты из проектного файла 'almai.yaml', а затем из командной строки.
После того, как обработаны все заданные препропмпты и все зависимости удовлетворены, производится генерация результата.

Значение поля requires для конкретного типа препромптов может быть YAML-списком. В этом случае этот список интерпретируется, как значения,
объединённые по AND.

Если все зависимости текущего препромпта имеют тот же тип, что и текущий препромпт, то можно использовать простой список или строковое значение.
Например:

### Пример frontmatter без указания типов препромптов, от которых зависит текущий препромпт
```
requires: [cpp-dev|c-dev,java-dev]
```

или

```
requires: 'cpp-dev|c-dev,java-dev'
```

Если используется отображение, допустимо указывать значение requires в виде YAML-списка, элементы которого могут содержать '|'.
Например:
```
requires:
  roles:
    - cpp-dev|c-dev
    - java-dev
```


# Поиск корневого каталога проекта

Создаваемая утилита (назовём её 'almai') может быть запущена из произвольного каталога проекта.
Она сканирует каталоги, начиная с текущего каталога, и ищет файл с именем 'almai.yaml'.
Каталог в котором находится данный файл, является корневым каталогом проекта.
Препромпты, которые относяться к проекту, будут располагаться в подкаталоге '.preprompts.almai',
или просто '.preprompts' (но данное имя каталога может использоваться и какими-нибудь другими системами/программами).
Каталог препромптов проекта добавляется последним, и значит, что он обрабатывается прежде всего.

Каталог '.preprompts' может использоваться каким-либо другими системами, поэтому, если существует
каталог '.preprompts.almai', который однозначно ассоциируется с нашей программой, каталог '.preprompts' игнорируется.


Подъем по файловой системе производится, пока
- не достигнут корень файловой системы
- не найден файл 'almai.yaml'
- не обнаружен один из каталогов '.git', '.out', '.vscode', '.build', 'build' - наличие этих каталогов косвенно указывает на корень проекта.
  В данном случае корень проекта считается найденным. Данная ситуация обрабатываетс ятак же, как ситуация с пустым проектным файлом.

Точный алгоритм поиска корневого каталога: 
утилита начинает поиск с текущего рабочего каталога и поднимается вверх по иерархии. В каждом каталоге выполняется следующая последовательность действий:
- Проверяется наличие файла almai.yaml. Если файл существует, данный каталог считается корневым каталогом проекта, и поиск прекращается.
- Если файл almai.yaml отсутствует, проверяется наличие любого из каталогов-маркеров: .git, .out, .vscode, .build, build. 
  Если хотя бы один из них присутствует, поиск останавливается, и проект считается найденным (с пустым файлом almai.yaml).
- Если ни almai.yaml, ни маркерных каталогов нет, утилита переходит в родительский каталог и повторяет шаги 1–2.

Подъём продолжается до тех пор, пока не будет достигнут корень файловой системы. Если корень достигнут, а проект не обнаружен, корневой каталог считается неопределённым.

# Содержимое файла almai.yaml

Файл проекта 'almai.yaml' должен содержать:
- preprompt-types - добавление (или замена) новых типов препромптов в проект, аналогично опции `--preprompt-types`, обрабатывается до опций командной строки,
  использование символа '-' (минус) - очищает список (работает так же, как и в командной строке) - таким образом, при 
  необходимости, мы можем полностью заменить преднастроенные в конфигурационных файлах типы препромптов;
- project-preprompts - препромпты, используемые на проекте по умолчанию, без явного их задания в командной строке, обрабатывается до опций командной строки;
- project-descriptions - соответствует опции командной строки `--project-descriptions`, работает аналогично preprompt-types.
- lang - язык промптов проекта.

Файл проекта 'almai.yaml' ищется и обрабатывается после обработки "встроенных" конфигурационных файлов, но до обработки опций и конфигурационных
файлов, заданных в командной строке.

Если мы хотим в проекте отбросить всю систему препромптов, которая была настроена в конфигурационных файлах, то аналогично
опции командной строки мы можем использовать символ минуса. Например:

```
preprompt-types: "-,roles,constraints" 
```

или 

```
preprompt-types: ["-",roles,constraints]
```

Аналогично работает параметр project-descriptions:
```
project-descriptions: ["-", "PROJECT", "REQUIREMENTS"]
```

или 
project-descriptions: "-,PROJECT,REQUIREMENTS"
```



## lang

Задаёт язык промптов проекта. По возможности используются промпты на данном языке, и только после этого - промпты по умолчанию.

Язык может быть задан:
- в файле проекта 'almai.yaml' (поле lang);
- через опцию командной строки `--lang=ru` (приоритет выше).

Если язык не задан, то используется язык, задаваемый опцией `--default-lang=...`.

Приоритет способов задания языка:
- опция командной строки --lang=ru (наивысший приоритет), также должна работать короткая версия `-l`.
- поле lang в файле проекта almai.yaml.
- опция --default-lang=xx (если указана).
- встроенное значение по умолчанию – en.

Если препромпт нужного типа не находится в локализованной подпапке в хранилище препромптов, он ищется без учёта языка.
Таким образом, если у нас (каким-либо способом из вышеперечисленных) задан язык ru, и мы хотим использовать role препромпт cpp-dev, проверяется наличие файла:
- 'preprompts/roles/ru/cpp-dev.md'
- 'preprompts/roles/cpp-dev.md'

Данное относительное имя проверяется во всех путях поиска. Если в более приоритетном пути 
есть 'roles/ru/id.md', он будет взят, даже если в менее приоритетном есть 'roles/id.md'.


## Локализация строк и заголовков.

Для локализации строк и заголовков используется текущий язык, заданный, как описано в предыдущем разделе.
У нас есть словарь словарей локализаций. В словаре верхнего уровня ключём является язык, во вложенных словарях ключём является
идентификаторы строк.

Для заголовков разделов используются ключи вида ТИП_ПРЕПРОМПТА-title, например: roles-title, constraints-title.
Ключи для заголовков формируются динамически на основании имени типа препромпта во множественном числе с добавлением суффикса '-title'.

Для раздела с разделами описания проекта, если они добавляются, следует использовать идентификатор project-descriptions-section-title.

Для настройки локализации используется опция командной строки `--translation=ЯЗЫК:ИД_СООБЩЕНИЯ:ТЕКСТ_СООБЩЕНИЯ`, например:
`--translation=ru:roles-title:Твоя роль` или `--translation=ru:project-descriptions-section-title:Описание проекта, его структура`.
Последний сегмент после второго двоеточия является текстом. Даже наличие там двоеточия ничего не ломате, и позволяет использовать текст,
содержащий двоеточия.

Если локализованный текст сообщения не найден, используется идентификатор типа препромпта (для препромптов) с первой заглавной буквой, 
например, для типа препромпта roles должен использоваться заголовок "Roles". Для заголовка секции с описанием проекта строка заголовка 
по умолчанию будет "Project Description".

В текущей версии нет никакой базы локализаций, все локализации добавляются из командной строки опцией `--translation`.
Для этого будут использоваться конфигурационные файлы опций, считываемые при запуске утилиты.

Опция `--translation`, как и все остальные опции, может быть указана как в командной строке непосредственно, 
так и в файлах опций.


## Кодировки файлов

Кодировка всех файлов предполагается одна - UTF-8.
Если происходит ошибка чтения UTF-8, то это вызывает предупреждение/ошибку.

Требуется правильно обрабатывать UTF BOM. Так как поддерживается только UTF-8, то валидна только UTF-8 BOM, другие типы BOM должны вызывать 
предупреждение/ошибку и такой файл должен игнорироваться, как будто его не существует.

При обнаружении BOM файл считывается с её учётом, а сам BOM-маркер удаляется до всей дальнейшей обработки.

При выводе следует использовать кодировку UTF-8 без BOM. Это относится как к выводу результатов, так и к выводу сообщений об ошибках.


## Вывод программы

Вывод программы выводится в STDOUT, если не указана опция `--output` (`-o`).

Всё сообщения об ошибках и предупреждениях выводятся в STDERR, в формате, который совместим с форматом вывода сообщений
об ошибках и предупреждений компилятора C++ GCC.

Если нет возможности узнать, в каком месте во входных файлах произошла ошибка, информацию о местоположении ошибки в файле
выводить не требуется.


## project-preprompts

`project-preprompts` представляет список списков типов препромптов и их идентификаторов.
На первом уровне - список типов препромптов, каждый из которых содержит список идентификаторов препромптов.

К этому списку добавляются препромпты, задаваемые из командной строки. Если препромпты повторяются, то оставляетс ятолько первое использование.
Это же относитс яи к добавлению препромптов из командной строки.

Если используется опция `--no-project-preprompts`, то список препромптов проекта не используется. Если в командной 
строке также ничего не задано, то при создании финального промпта никакие препромпты не будут использоваться,
и следует выдать предупреждение. Если не найден ни один из файлов опции `--project-descriptions` ('PROJECT.md' и тп) и нет препромптов - 
то это фатальная ошибка - мы не знаем, что генерировать.

На YAML использование project-preprompts в проектном файле выглядит примерно так:

```
project-preprompts:
  roles: [cpp-dev, devops]
  objectives: [analyze]
  constraints: [cpp17]
```

или так:

```
project-preprompts:
  roles: cpp-dev, devops
  objectives: analyze
  constraints: cpp17
```

То есть, тут нужно отображение ключа в последовательность, но для простоты записи конфигурационных надо уметь 
записывать проеобразовывать второй вариант (отображение ключа в строку) в первый вариант, и внутри программы должно
использоваться отображение ключа в последовательность.


# Вставка препромптов в результирующий документ

Препромпты могут быть простыми и расширенными.

В процессе сборки простые и сложные препромпты группируются раздельно для каждого из типов препромптов.

При генерации результирующего препромпта сначала выводятся простые препромпты, затем - расширенные.

Перед вставкой как простых препромптов выводится вводный текст раздела (отдельным параграфом).
Вводный текст зависит от типа препромптов в генерируемом разделе.

Для вводного текста (introductory text) используются идентификаторы вида: 
'ТИП_ПРЕПРОМПТА-intro-simple' и 'ТИП_ПРЕПРОМПТА-intro-extended', например,
roles-intro-simple, roles-intro-extended, constraints-intro-simple, constraints-intro-extended.
Ключи для вводного текста формируются динамически на основании имени типа препромпта во 
множественном числе с добавлением суффиксов '-intro-simple' и '-intro-extended'.

Если для текущего типа препромпта локализации соответствующего вводного текста не находится,
то вводный текст не используется.

Механизм локализации описан в разделе "Локализация строк и заголовков".

В метаданных препромпта может быть явно указано, что препромпт является простым или расширенным (атрибут extended).
Также может быть указан режим автоматического определения типа препромпта - auto. 
В этом случае текст препромпта следует проверить на наличие заголовков - если их нет, то препромпт является простым,
иначе - расширенным.


# Простые препромпты

Простой препромпт представляет из себя один или несколько параграфов, и, возможно, нумерованные и ненумерованные списки.
Простой препромпт не содержит заголовков, листингов, и тп.

Алгоритм вставки простых препромптов следующий:
- Если в разделе только один препромпт, он вставляется как есть.
- Если в проекте несколько препромптов, то они каждый обрабатываются так:
  - все пустые строки в начале и в конце удаляются (пустые строки - строки, не содержащие ничего, кроме пробелов). В середине - оставляются, как есть.
  - для первой строки добавляется префикс "- " - создаёт элемент ненумерованного списка;
  - для последующих строк добавляется префикс "  " (два пробела) - продолжение пункта списка. Если в препромпте
    есть список, то он автоматически становится вложенным.

В простых препромптах, если в тексте препромпта встречается более одной пустой строки подряд, лишние пустые строки
следует удалить и оставить только одну пустую строку вместо любого количества пустых строк.

Для множественных препромптов одного типа, таким образом, текст препромпта становится текстом элемента списка первого уровня,
а список в препроме, если он задан, становится элементов списа второго уровня.

Если простой препромпт содержит только ненумерованный список (без вводной фразы), это должно вызывать фатальную ошибку: 
основное в препромпте - 
это текст в виде одного или нескольких абзацев, а списки могут лишь расширять и дополнять основной текст препромпта.

Данный алгоритм вставки препромптов должен использоваться всегда, вне зависимости от того, каким образом был добавлен препромпт.

Формализованный алгоритм объединения препромптов одного типа:
Для формирования вложенных списков при множественных препромптах используется простое механическое добавление двух пробелов 
в начало каждой строки препромпта, кроме первой. Это преобразование не требует анализа содержимого: markdown-список, 
находящийся в тексте препромпта, автоматически становится вложенным благодаря добавленным отступам. Например:

```
- Основной пункт (текст препромпта)
  - Вложенный пункт из препромпта
```

# Расширенные препромпты

Расширенные препромпы - это препромпты, которые могут содержать разделы и подразделы, а также листинги, 
и прочие структурные элементы целого документа.

Расширенные препромпы обрабатываются вставкой всего содржимого препромпта с понижением уровня заголовков на один уровень.

Перед вставкой расширенного препромпта из него удаляются ведущие и завершающие пустые строки (строки, не содержащие ничего, кроме пробелов).

## Понижение уровня заголовка

Понижение уровня заголовка на один уровень в markdown - это простое добавление символа '#' к уже существующим.
Если заголовок имеет уровень 6 (начинается с "######") - то его уровень не понижается, а заголовок преобразовывается
в отдельный параграф, выделенный жирным (два знака '*' перед текстом и после). Таким образом, хотя структура
документа немного меняется (пропадает заголовок самого нижнего уровня), но мы не получаем возможную ошибку, 
если бы сделали заголовок уровня 7, и оставляем визуальное выделение бывшего заголовка.

Примечание: заголовки в препромптах для простоты поддерживаются только в ATX-стиле, заголовки Setext-стиля не поддерживаются.


# Описание структуры проекта

Описание структуры проекта обычно хранится в файле 'PROJECT.md' (конкретные имена файлов с описаниями различных аспектов проекта,
добавляемые в препромпт, задаются опцией `--project-descriptions`).
Это произвольное описание проекта, которое добавляется
к создаваемому промпту. 

Файл 'PROJECT.md' может содержать YAML frontmatter, но его содержимое игнорируется.

Файл 'PROJECT.md' вставляется по правилам расширенных препромптов, но без вводного текста.


# Прочие специальные файлы описаний

Файл 'PROJECT.md' является специальным файлов описаний. Может быть некоторое количество
подобных файлов, кроме 'PROJECT.md', например: 'REQUIREMENTS.md', 'ROADMAP.md', 'STATE.md'.

Опция `--project-descriptions` задаёт имена файлов, которые надо включить в результирующий документ,
а также порядок их включения. Расширение '.md' не указывается. Например:
`--project-descriptions=PROJECT,REQUIREMENTS,ROADMAP,STATE`.

Файлы описаний могут содержать YAML frontmatter, но его содержимое игнорируется.

Файлы описаний вставляется по правилам расширенных препромптов, но без вводного текста.

Если файл пуст (содержит только пустые строки - пробелы и переводы строки) - он не вставляется.
Файл описаний вставляется всегда, если такой файл существует и содержит не только пустые строки.

Опция `--project-descriptions` работает аналогично опции `--preprompt-types` - так, символ минус очищает содержимое списка,
а параметр 'project-descriptions' в проектном файле работает аналогично параметру 'preprompt-types'.

Файлы описаний проекта ищуться в том же каталоге, в котором расположен файл проекта 'almai.yaml'.

Если в том же каталоге, в котором расположен файл проекта 'almai.yaml' существует каталог 
'.project-descriptions' или '.project-descriptions.almai', то используются файлы описаний, которые расположены в этих каталогах.

Если каталоги '.project-descriptions' и '.project-descriptions.almai' существуют одновременно, то используется каталог
'.project-descriptions.almai', а каталог '.project-descriptions' игнорируется.

Файлы, имена которых перечисляются в опции `--project-descriptions` (без расширения .md) должны быть полноценными markdown документами
со своими заголовками и своей структурой.



# Структура каталогов дистрибутива утилиты almai

```
almai/                          # корневая папка дистрибутива
├── bin/                        # двоичные исполняемые файлы, а также исполняемые скрипты
│   └── ...
├── conf/                       # конфигурационные файлы
│   ├── almai.options           # Основной конфигурационный файл (поставляется в составе дистрибутива)
│   ├── almai.custom.options    # Системный конфигурационный файл (может отсутствовать, настраивается администратором системы, предназначен для всех пользователей системы)
│   └── ...
├── preprompts/                 # Базовые препромпты (поставляются в составе дистрибутива)
│   └── ...
├── preprompts.custom/          # Системные препромпты (могут отсутствовать, настраиваются администратором системы, предназначены для всех пользователей системы)
│   └── ...
│   
└── (другие каталоги, например, документация)
```

В системах Windows обычно путь к каталогу bin прописывается в переменную окружения PATH, в Linux и прочих *nix
практикуется другой способ - в каталоге '/usr/bin' или '/usr/local/bin' создаётся символическая ссылка 'almai.py' на
'/usr/local/share/almai/bin/almai.py'.
Корневым каталогом дистрибутива в данном случае всё равно будет считаться '/usr/local/share/almai'.


Пользовательский конфигурационный файл располагается в каталоге `%HOME%/.almai` (`$(HOME).almai`) и носит имя `%HOME%/.almai/.options` (`$(HOME)/.almai/.options`).
Пользовательские препромпты располагаются в каталоге `%HOME%/.almai/preprompts` (`$(HOME)/.almai/preprompts`).

При запуске утилиты производится обработка конфигурационных файлов в следующем порядке:
- Сначала обрабатывается основной конфигурационный файл (`almai.options`), если существует, 
- затем настраиваемый системный конфигурационный файл (`almai.custom.options`),
- затем пользовательский конфигурационный файл (`%HOME%/.almai/.options`).

Обработка опций, заданных в командной строке, и конфигурационных файлов, заданных в командной строке с использованием символа '@', производится
после обработки вышеописанных "встроенных" конфигурационных файлов. Файл опций, задаваемый через '@', обрабатывается точно также, ка ки конфигурационные файлы, 
предопределённые по умолчанию.

Если в файле опций встречается опция, начинающаяся с '@', то это вложенное подключение командного файла. При обработке командных файлов через символ '@' следует 
предотвращать циклы, и выдавать предупреждение/ошибку при их обнаружении.

Можно исключить из обработки каждый из этих файлов. Для этого следует предусмотреть опции командной строки:
- `--no-builtin-options` - не обрабатывать основной конфигурационный файл;
- `--no-custom-builtin-options` - не обрабатывать настраиваемый системный конфигурационный файл;
- `--no-user-builtin-options` - не обрабатывать пользовательский конфигурационный файл.


# Формат конфигурационных файлов

Конфигурационные файлы могут содержать коментарии. Коментариями являются строки, начинающиеся с символа '#' (решетка) в начале строки (допустимы ведущие пробелы).

Пустые строки игнорируются.

В значимых строках должны располагаться опции, такие же, как указываемые в командной строке, по одной опции на строку - строка не разбивается на опции, а обрабатывается , как одна опция.
Переносы не допустимы. Пробелы не нужно экранировать или заключать в кавычки.

При обработке опций должны поддерживаться макросы вида: '$(ИМЯ_ПЕРЕМЕННОЙ)'.

Некоторые макросы с рзличными именами могут означать одно и то же. Они являются псевдонимами/алиасами и используются по историческим причинам. 
Возможные имена макросов (и их значение):
- AppName - имя исполняемого файла (скрипта) приложения, без - `almai`;
- AppPath, AppRootPath, AppRoot - путь к корневой папке инсталляции;
- AppBinPath, AppBin - путь к исполняемым файлам (скриптам);
- AppConfPath, AppConf - путь к конфигурационным файлам;
- UserHome - путь к домашнему каталогу пользователя;
- UserAppPath - путь к каталогу настроек программы в домашнем каталоге пользователя (`%HOME%/.almai/`);
- ProjectRoot - устанавливается, если обнаружен проект ('almai.yaml'), но на момент обработки встроенных конфигов он не определён (пустая строка);
- CWD – текущий рабочий каталог.

Макросы раскрываются только при обработке конфигурационных файлов опций (включая файлы, переданные через @) и нигде более.

На момент обработки конфигурационных файлов (до поиска проекта) ProjectRoot неизвестен. Он должен раскрываться в пустую строку.

Если встречается неизвестный макрос, следует остановится по фатальной ошибке.

**Замечание по поводу макроса AppRoot (и псевдонимов)**
Предположим, исполняемый скрипт находится в '/usr/local/bin/almai' (наверное, сделан симлинк),
а каталог дистрибутива — '/usr/local/share/almai'. Значение макроса AppRoot в этом случае должно быть '/usr/local/share/almai' -
указывать на корневой каталог дистрибутива программы.


# Обработка препромптов

Препромпты обрабатываются в порядке, обратном порядку их указания. Путь к каталогу с препромптами задаётся 
опцией командной строки `--preprompts-dir=...` (такая же опция может быть задана в конфигурационном файле).

Приоритет каталогов определяется порядком их добавления в список поиска: последний добавленный каталог имеет наивысший приоритет (проверяется первым). 

Например, в 'almai.options' есть строки:
```
--preprompts-dir=$(AppConf)/preprompts
--preprompts-dir=$(AppConf)/preprompts.custom
--preprompts-dir=$(UserAppPath)/preprompts
```

Также в пути поиска препромптов автоматически добавляется '$(ProjectRoot)/.preprompts.almai' (если он существует) и 
'$(ProjectRoot)/.preprompts' (если он существует, но каталога '$(ProjectRoot)/.preprompts.almai' не существует).
"Макрос" '$(ProjectRoot)' тут обозначает корневой каталог проекта, в котором находится файл 'almai.yaml'. Добавление
этих каталогов производится после обработки встроенных конфигурационных файлов, но до обработки опций командной строки.
Пути поиска препромптов, заданные в командной строке опцией `--preprompts-dir` (или в командном файле, указанном в командной строке)
добавляются в конец списка и имеют наивысший приоритет (обработка идёт в порядке от хвоста к началу).

Тогда поиск препромптов будет производится в следующих каталогах:
- '$(ProjectRoot)/.preprompts.almai' (или '$(ProjectRoot)/.preprompts');
- '$(UserAppPath)/preprompts';
- '$(AppConf)/preprompts.custom';
- '$(AppConf)/preprompts'/


# Обработка файлов опций

Если в командной строке аргумент начинается с символа '@' - это значит, что передан файл опций, и его надо построчно обработать, 
как будто опции заданы непосредственно в командной строке. Это позволяет быстро и удобно переиспользовать наборы опций.


# Задание генерации промпта

## Основные принципы

Предполагается, что типы препромптов могут добавляться пользователем. Порядок следования задаётся настройками:
опцией `--preprompt-types=roles,objectives,contexts,formats,extras,constraints`. В данном случае список типов препромптов будет дополнятся.

Опция `--preprompt-types` задаёт не только порядок следования разделов, но и задаёт, какие типы препромптов могут быть использованы.

Опция `--preprompt-types` является накопительной, аддитивной - она добавляет новые значения к существующему списку. Минус ('-') используется для 
того, чтобы очистить ранее созданный список типов препромптов и начать задавать свою систему препромптов. 

Каждый раз, когда список типов препромптов модифицируется опцией `--preprompt-types` (или параметром 'preprompt-types' в файле проекта 'almai.yaml'),
следует перестраивать набор опций для разбора аргументов командной строки.

Препромпты типов, перечисленные в данной опции, будут искаться в соответствующих каталогах в базах препромптов 'preprompts'.

Для всех типов препромптов, заданных данной опцией, должна динамически добавлятся обработка соответствующих 
опций командной строки, так, если у нас задана опция `--preprompt-types=roles,objectives`, то программа должна 
обрабатывать опции `--roles`/`--role` и `--objectives`/`--objective` (в единственном и множественном числе),
а также опции `--role-text` и `--objective-text` - для непосредственного задания препромпта текстом
(в имени опции используется только единственное число).

Если тип не перечислен, соответствующая опция не должна обрабатываться, т.е. если `--preprompt-types=...` нет типа roles, то указание
опций `--role`/`--roles` будет давать предупреждение или ошибку.

Пользователь может захотеть построить свою собственную систему типов препромптов. Для этого ему надо удалить все ранее
заданные препромпты. Это достигается указанием символа минус ('-') вместо одного из имён типов препромптов (обычно, первого в списке).
Пример: `--preprompt-types=-,roles,objectives,contexts,formats,extras,constraints` - очищается список типов препромптов,
затем в список добавляются типы припромптов roles, objectives, contexts, formats, extras, constraints.

Если `--preprompt-types` пуст (не настроен, или был очищен), то следует выдавать предупреждение (или ошибку в strict режиме).
Если 'PROJECT.md' отсутствует, то это будет фатальная ошибка.

Список препромптов может быть также модифицирован в файле проекта 'almai.yaml'.
На YAML это выглядит примерно так: 

```
preprompt-types: cpp-dev, devops
```
или 
```
preprompt-types: [cpp-dev, devops]
```
Данная опция в файле проекта работает аналогично опции командной строки `--preprompt-types`.


Опция `--preprompt-types` не только задаёт порядок включения препромптов в результирующий документ, она также
определяет, какие типы препромптов мы обрабатываем, и указывает обрабатывать соответствующие опции командной строки.
Для приведённого примера мы должны уметь обрабатывать следующие опции командной строки:
- --role/--roles
- --objective/--objectives
- --context/--contexts
- --format/--formats
- --extra/--extras
- --constraint/--constraints

Об обработке единственного/множественного числа рассказывается ниже.

Опции --role/--roles - обрабатываются одинаково, --objective/--objectives - также обрабатываются одинаково, и тд.

Также, для обработки непосредственно задаваемых текстов следует добавить опции:
- --role-text
- --objective-text
- --context-text
- --format-text
- --extra-text
- --constraint-text

Текстовые препромпты добавляются как текст, и не использут механизм зависимостей. 
Не предполагается использовать текстовые препромпты для задания многострочного текста.
Многострочный текст нельзя передать в качестве текстового промпта.

Препромпты добавляются каждый в свой раздел, не должно быть такого, что у нас в результирующий документ попали 
препромпты roles, потом препромпты objectives, а потом опять следуют дополнительные препромпты roles.

Порядок следования препромптов в соответствующем разделе соответствует порядку их указания в командной строке и/или
в файле проекта.

В качестве имени типа препромпта может использоваться слово как в единственном, так и во множественном числе.
Внутренее представление - всегда только множественное число.
Каталоги, в которых располагаются файлы препромптов, всегда носят имя во множественном числе: roles, objectives, formats.


Правила отображения единственного числа на множественное таковы:
- Есть словарь, в котором хранится отображение: ЕДИНСТВЕННОЕ_ЧИСЛО -> МНОЖЕСТВЕННОЕ_ЧИСЛО. Например: analysis -> analyses.
- Также есть словарь, в котором хранится отображение МНОЖЕСТВЕННОЕ_ЧИСЛО -> ЕДИНСТВЕННОЕ_ЧИСЛО. Например: analyses -> analysis.
- Пары слов в словари задаются опцией командной строки: `--dict=ЕДИНСТВЕННОЕ_ЧИСЛО:МНОЖЕСТВЕННОЕ_ЧИСЛО`. Например: `--dict=analysis:analyses`.
  По этой команде обновляются оба словаря.
- Преобразование во внутренее представление (множественное число):
  - если слово в словаре (ЕДИНСТВЕННОЕ_ЧИСЛО -> МНОЖЕСТВЕННОЕ_ЧИСЛО) есть, то используется значение по ключу.
  - если слова в словаре нет, и оно заканчивается на букву 's' - это множественное число и ничего делать не требуется. 
  - иначе преобразуем во множественное число добавлением буквы 's'. Примеры: role/roles, objective/objectives, context/contexts.
- Преобразование из множественного числа в единственное (для создания опций командной строки в единственном числе):
  - если слово в словаре (МНОЖЕСТВЕННОЕ_ЧИСЛО -> ЕДИНСТВЕННОЕ_ЧИСЛО) есть, то используется значение по ключу.
  - если слова в словаре нет, и оно заканчивается на букву 's' - это множественное число, удаляется конечная 's'.
  - иначе ничего делать не требуется.

Возможные ошибки. Например, если у нас в словари не добавлена пара analysis -> analyses, то программа попытается получить из слова analyses единственное
число отбрасыванием завершающей 's' и получится "analyse", но это не верно, правильно - "analysis". Такие случаи оставляются на усмотрение пользователя,
он сам должен добавить словарные пары, если использует имена типом препромптов с нестандартным образованием множественного числа.

Вышеуказанный алгоритм является достаточно полным, с учетом использования словарей. Не требуется применять какие-то ещё алгоритмы или эвристики.

Преобразование МНОЖЕСТВЕННОЕ_ЧИСЛО -> ЕДИНСТВЕННОЕ_ЧИСЛО требуется для того, чтобы динамически добавлять опции конмадной строки. 
Так, для типа препромпта roles должны быть добавлены и обработаны опции `--roles` и `--role`.


# Универсальная короткая опция добавления препромпта

Каждый раз укаывать тип препромпта довольно многословно, и нужно добавить более простой способ их указания.

Добавим опцию `--preprompt` (кортокий вариант - `-p`). 
Использование выглядит так: `--preprompt=ИМЯ_ПРЕПРОМПТА`. Видно, что тут не указывается тип препромпта.

Данная опция должна работать так: 
Для каждого типа препромптов из текущего списка 
preprompt-types проверяется наличие файла с именем ИМЯ_ПРЕПРОМПТА.md в соответствующем каталоге (с учётом языка).
Если локализованной версии не обнаруживается , проверяется наличие language neitral версии.
Если файл существует, тип препромпта добавляется в список кандидатов.

Идея в том, чтобы собрать все активные типы препромптов, в которых доступен заданный в опции препромпт, и потом сделать вывод,
есть ли препромпт с таким идентификатором вообще, и существует ли он в единственном варианте, для возможности однозначной 
его идентификации только по идентификатору, без указания типа препромпта.

Если по окончании поиска в списке нет элементов (типов препромптов), то выдаётся предупреждение 
(или ошибка, в зависимости от strict режима), о том, что препромпт не найден. 

Если по окончании поиска в списке типов препромптов находится только один элемент, то работает
стандартный алгоритм добавления препромпта с учётом найденного типа препромпта.

Если по окончании поиска в списке типов препромптов находится больше одного элемента,
то выводится сообщение об ошибке (всегда, без учёта опции strict) о том, что найдено несколько вариантов,
при этом выводится список типов препромптов, где был найден препромпт с данным именем,
и программа завершается.

Значения в опции `--preprompt` могут быть заданы как непосредственно в одной опции через запятую, так и в нескольких по отдельности,
результат должен быть одинаков. То же самое относится и к динамическим опциям с именами типов препромптов.


# Вывод сведений об известных препромптах

Для вывода сведений об известных препромптах созданим команду `--list`. Команда `--list` без аргументов равнозначна команде `--list=all`.
Команда `--list=all` выводит списки препромптов всех типов.
Команда `--list=roles,objectives` - выведет список доступных препромптов в категориях roles и objectives.

Сканирование на предмет сущствования препромптов осуществляется по всем заданным preprompt-каталогам, для всех возможных языков -
рекурсивно производится поиск файлов '*.md' для каждого типа препромпта, и обрезается расширение - получаем идентификатор препромпта. 
Для каждого типа препромпта составляется набор найденных идентификаторов препромптов данного типа.

Далее делается фильтрация по запрошенным типам препромптов, и производится вывод доступных препромптов.

Для каждого препромпта выводится его идентификатор, например, cpp-dev, python-dev, и описание (из атрибута title, заданного во frontmatter соответствующего файла).
Описание может отсутствовать.

Вывод сведений о препромптах производится с учётом текущего языка: с начала ищутся все локализованные версии препромптов, затем к ним добавляются те, 
которые не используют механизм локализации. Если есть локализованный препромпт, то вся информация берётся только из него, и только. Если чего-то не хватает,
например, описания, то оставляем описание пустым.

Если производится вывод для нескольких типов препромптов, то вывод надо соответственно разбить на группы, задав для кадой группы соответствующий заголовок.

Следует предусмотреть вывод подробной информации, с выводом полного имени файла препромпта. Для этого используем аналогичную опцию `--verbose-list`.

Вывод желательно делать в виде текстовой таблицы, с учетом максимальной длины идентификаторов + 2. Заголовки типов препромптов (групп) прижаты к левому краю,
отдельные строки сдвинуты на два символа:
```
roles:
  c-dev              C programmer
  cpp-dev            C++ programmer
  cpp-programmer     C++ programmer
  python-programmer  Python programmer

objectives:
  analize            Analyze project structure
```

При выполнении команды --list (с необязательным указанием типов) отображаются препромпты, доступные в данный момент с учётом текущего языка 
(`--lang`, 'lang' из 'almai.yaml' или `--default-lang`). Для каждого идентификатора показывается та версия, которая была бы загружена при генерации промпта:
- если в путях поиска существует локализованная версия (например, roles/ru/cpp-dev.md), берётся информация (title, description) именно из неё;
- если локализованной версии нет, используется базовая (roles/cpp-dev.md);
- если файл отсутствует вовсе, идентификатор не отображается.

Таким образом, список всегда отражает реально доступные для использования препромпты с учётом языка.


# Технические требования

##  Общая архитектура и модули

Важно, чтобы программа состояла из определенных логических блоков для облегчения ревью и доработки, 
но детали реализации внутри этих блоков не задаются жестко и оставляются на усмтрение разработчика.

Желательно (но не обязательно) использовать ООП и классы, и разные модули разнести по разным файлам, которые включаются в основной файл скрипта.

Желаельно использовать модульную структуру с чёткими интерфейсами:
- ConfigManager – загрузка конфигураций (файлы, CLI, макросы).
- PrepromptLoader – поиск и чтение файлов с учётом языка и приоритетов.
- DependencyResolver – разрешение зависимостей с проверкой циклов.
- PromptAssembler – сборка итогового Markdown.
- TokenCounter – подсчёт токенов (эвристика + внешние библиотеки).
- CLI – обработка аргументов (рекомендуется argparse или click).


## Версия Python
Версия Python должна быть не ниже 3.7, но не хотелось бы фиксироваться на какой-то конкретной подверсии, надо сделать максимально совместимо.

## Дополнительные библиотеки

По возможности следует использовать только встроенные библиотеки. 
Для обработки YAML можно использовать библиотеку pyyaml.
Если библиотека pyyaml не установлена, то следует выдать сообщение с информацией по её установке.

## Платформенная совместимость

Для определения домашнего каталога пользователя следует использовать встроенные в Python средства, а не системные пременные (которые могут быть разными или отсутстворвать).
%HOME% ($HOME) везде приводилось для примера (в Windows нет специального обозначения ~/тильда для домашнего каталога).
Не нужно ориентироваться на переменные %HOME% или %USERPROFILE% вручную.

Для работы с путями предпочтительно использовать pathlib.

Учитывать, что на Windows разделитель пути – обратная косая черта, но в конфигах можно использовать прямую, Python преобразует.
С другой стороны, Windows поддерживает и прямой слэш.
В конфигурационных файлах и везде при использовании путей под Windows можно использовать как прямой слэш, так и обратный, под Linux - только прямой.


## Аргументы опций командной строки

Несколько идентификаторов для одной опции могут быть заданы как в одной опции, так и в нескольких: `--roles=cpp-dev,devops` или `--roles=cpp-dev --roles devops`.
Эти варианты равнозначны.


## Целевая ОС
Результирующий скрипт должен работать в операционных системах Windows и Linux, а также в Unix-подобных (например, FreeBSD).

## Обработка ошибок

Может быть два уровеня обработки ошибок: обычный и строгий. Строгий включается опцией '--strict[=0/1/false/true]'
и выключается опцией '--no-strict'. В обычном режиме выдаются предупреждения и обработка продолжается с пропуском ошибочной информации.
В строгом режиме обработка останавливается сразу после вывода сообщения об ошибке.
`--strict` без значения эквивалентен `--strict=true`.
Допустимые значения: true/false, 1/0, on/off, yes/no, с игнорированием регистра символов.
Опция --no-strict отключает строгий режим (переопределяет предыдущие настройки).

Что считать ошибкой:
- Отсутствие файла препромпта для указанного ID: в обычном режиме – предупреждение (пропуск), в строгом – ошибка и останов.
- Ошибка чтения текста в кодировке UTF-8 - в обычном режиме – предупреждение (пропуск), в строгом – ошибка и останов.
- Циклические зависимости: ошибка в любом режиме; требуется корректно обрабатывать транзитивные зависимости. Останов на первой ошибке, собирать все ошибки слишком сложно.
- Не найден PROJECT.md: предупреждение, если не требуется обязательное включение (но если проект не задан, а препромпты есть – можно продолжать).
- Нет ни одного препромпта (ни из CLI, ни из проекта) и нет PROJECT.md – фатальная ошибка (нечего генерировать).
- Ошибка чтения файла (кодировка, повреждение) – предупреждение/ошибка аналогично отсутствию.
- Синтаксическая ошибка в frontmatter – предупреждение (файл обрабатывается как обычный текст без frontmatter) или ошибка в строгом режиме.

Все предупреждения и информационные сообщения должны выводиться в stderr.

Если у нас strict режим, или встретилась ошибка, являющаяся фатальной в любом режиме, то обработка останавливается, результат не формируется, возвращается какой-то код ошибки.
Если у нас обычный режим, и мы выдали предупреждение, то обработка продолжается, и результирующий документ должен быть сформирован, 

Коды возврата: 0 – успех, 1 – ошибка (в т.ч. фатальная в строгом режиме), другие коды можно не использовать. 
Можно дополнительно определить: 1 – общая ошибка, 2 – ошибка в конфигурации, 3 – ошибка зависимостей и т.д. Это опционально.

## Подсчёт токенов

Утилита при генерации финального документа должна подсчитывать для него количество ИИ-токенов.
Точный подсчёт зависит от конкретной модели, и в общем случае не возможен. Требуется подсчитать приблизительное число токенов по словам и символам,
при этом, если слово (последовательность алфавитно-цифровых символов) заканчивается на цифры - это один токен, а если цифры идут в начале 
последовательности - то это два токена - число и слово. Если слово начинается и заканчивается на цифры, то первые цифры - один токен, а 
последующие цифры - второй токен. Пример: "48м2" разбивается на два токена - число 48 и строка "м2" (метры квадратные).

Буквами являются символы не только английского алфавита, но также и кириллицы, греческого, иврита, арабского языков.

Последовательности из одинаковых символов считай также за один токен.

Остальные символы считаются каждый за отдельный токен.

**Формальное описание эвристического алгоритма**
- Текст разбивается на последовательности символов.
- Любая последовательность букв (включая кириллицу, греческие, иврит, арабские и т.д. - Unicode-буквы) считается одним токеном.
- Любая последовательность букв (включая кириллицу, греческие, иврит, арабские и т.д. - Unicode-буквы), которая продолжается арабкими цифрами (ASCII), считается одним токеном.
- Любая последовательность цифр считается одним токеном.
- Если цифры и буквы чередуются, они разделяются: часть из цифр -> токен-число, остальная часть -> токен-слово. Например, «48м2» → [«48», «м2»].
- Дефис (юнкодный, но не символ минус), если слева и справа от него любые буквы - считается частью токена, а не разделителем. 
- Символ минус - всегда считается разделителем токенов, кроме случаев, когда слева или справа от него находятся любые буквы, 
  кроме латинских: "abc-def" - три токена, "кто-нибудь" - один токен, "BNF-грамматика" - один токен.
- Апостроф, если слева и справа от него латинские буквы - считается частью токена, а не разделителем.
- Последовательности одинаковых символов пунктуации или операторов (например, '---') считаются одним токеном.
- Последовательности символов: '--', '++', '-=', '+=', '==', '!=', '<=', '>=', '<=>', '->', '=>', '::' - считаются одним токеном (стандартные операторы в ЯП).
- Все остальные одиночные символы (знаки препинания, пробелы, скобки и т.п.) считаются отдельными токенами.

Результат (число токенов) выводится в stderr с пометкой [ heuristic token count: N ].

Количество токенов надо вывести в STDERR.

Если в системе установлена библиотека tokenizers, rs-bpe или smolbpe, подсчитай количество токенов при помощи первой найденной из них 
и также выведи в STDERR, указав, какая библиотека использовалась.

Если библиотеки для подсчёта токенов требуют каких-то параметров (например, тип модели), 
то следует указать наиболее подходящее значение параметров для данной библиотеки.

При наличии установленных библиотек для токенизации утилита автоматически выбирает первую доступную из списка: 
tokenizers, rs-bpe, smolbpe. Для выбранной библиотеки используются следующие параметры по умолчанию, 
обеспечивающие приближение к токенизации современных LLM (например, GPT-семейства):
- Для tokenizers используется предобученная модель "gpt2" (если не указано иное в конфигурации).
- Для rs-bpe и smolbpe используются параметры, эквивалентные модели GPT-2/4, если библиотека поддерживает автоопределение; в противном случае — параметры по умолчанию, разумные для английского и многоязычных текстов.
- Если выбранная библиотека требует явного указания модели, но подходящая модель не найдена, утилита выдаёт предупреждение в STDERR и переходит к эвристическому подсчёту.

При отсутствии всех указанных библиотек используется только эвристический алгоритм.

Опция `--tokenizer-lib-list` выводит список доступных библиотек токенизации, или сообщение "tokenizer libraries not found".

Для использования конкретной библиотеки токенизации служить опция `--tokenizer-lib=ИМЯ_БИБЛИОТЕКИ`. 
Без данной опции используется первая найденная библиотека токенизации.


# Тесты

Для тестирования следует использовать библиотеку pytest.
Если библиотека pytest не установлена, то следует выдать сообщение с информацией по её установке.

Следует написать следующие тесты:
- Модульные тесты (unit tests) – проверка отдельных компонентов в изоляции.
- Интеграционные тесты – проверка взаимодействия компонентов и работы с реальной файловой системой.
- Тесты командной строки (CLI) – проверка обработки аргументов, опций, генерации вывода.


Необходимо написать тесты, покрывающие ключевую функциональность. Тесты должны быть написаны на **pytest**. 
Использовать встроенные возможности pytest: фикстуры, параметризацию, временные каталоги (`tmp_path`), 
моки (`pytest-mock` или встроенный `monkeypatch`). 
Для сложных сценариев с CLI можно использовать `CliRunner` из `click.testing` (если CLI построен на click) или `subprocess` с временными файлами.

## Требования к тестам
1. **Понятность** – каждый тест должен проверять одну конкретную функциональность.
2. **Изолированность** – тесты не должны влиять друг на друга; использовать временные файлы и директории.
3. **Покрытие** – по возможности стремиться к покрытию не менее 80% кода (можно проверить позже с pytest-cov).
4. **Параметризация** – для проверки нескольких однотипных сценариев использовать `@pytest.mark.parametrize`.
5. **Документирование** – каждый тест должен иметь docstring с описанием, что проверяется.

## Структура тестов
Рекомендуется организовать тесты по модулям/пакетам, например:
```
tests/
├── conftest.py              # общие фикстуры
├── test_loader.py           # загрузка препромптов
├── test_frontmatter.py      # парсинг YAML frontmatter
├── test_dependencies.py     # разрешение зависимостей
├── test_localization.py     # локализация
├── test_assembler.py        # сборка финального документа
├── test_config.py           # конфигурационные файлы и макросы
├── test_cli.py              # командная строка
├── test_integration.py      # интеграционные тесты
└── test_errors.py           # обработка ошибок в разных режимах
```

## Что нужно протестировать (детали)

### 1. Загрузка препромптов (`test_loader.py`)
- Фикстура: создавать временную структуру `.preprompts` с подкаталогами `roles`, `objectives` и т.д., а также с локализованными подкаталогами `ru`, `en`.
- Проверить, что файл ищется по всем путям в правильном порядке (приоритет: сначала проектные, потом пользовательские, потом системные).
- Проверить выбор локализованной версии: если язык `ru`, то должен браться файл из `roles/ru/cpp-dev.md`, а если его нет – из `roles/cpp-dev.md`.
- Проверить, что при отсутствии файла возвращается `None` (или выбрасывается исключение, зависит от реализации). Для удобства тестирования можно сделать функцию `load_preprompt(category, id, lang, search_paths)`, возвращающую содержимое или `None`.

### 2. Парсинг frontmatter (`test_frontmatter.py`)
- Фикстуры: временные файлы с различным содержимым.
- Проверить:
  - Файл без frontmatter – весь файл считается телом, метаданные пусты.
  - Файл с frontmatter (валидный YAML) – метаданные извлекаются, тело – остаток после `---`.
  - Файл только с frontmatter и закрывающим `---` – тело пусто.
  - Файл с незакрытым frontmatter (только открывающий `---`) – весь файл (включая открывающую строку) считается телом.
  - Поле `requires` со сложной строкой: парсинг должен возвращать структуру, представляющую логическое выражение (например, список условий). Упрощённо можно проверять, что для строки `"roles:cpp-dev|python-dev, constraints:cpp17"` функция `parse_requires` возвращает кортеж `(('roles', ['cpp-dev', 'python-dev'], 'OR'), ('constraints', ['cpp17'], 'AND'))` или подобное.
  - Ошибка YAML (например, некорректный отступ) – в обычном режиме выдать предупреждение и обработать как отсутствие frontmatter; в строгом – поднять исключение.

### 3. Разрешение зависимостей (`test_dependencies.py`)
- Функция для проверки: `resolve_dependencies(selected_ids, available_preprompts)`, где `selected_ids` – словарь {категория: [id]}, `available_preprompts` – функция, возвращающая метаданные препромпта по категории и id.
- Проверить:
  - Простая зависимость: A требует B – в результате добавляется B.
  - Транзитивная зависимость: A требует B, B требует C – в результате A, B, C.
  - Условие И/ИЛИ: A требует `(X OR Y) AND Z`. Случаи:
    - Есть X и Z – OK.
    - Есть Y и Z – OK.
    - Есть X, но нет Z – ошибка (предупреждение).
    - Нет ни X, ни Y – ошибка.
  - Циклическая зависимость: A→B→A – должно обнаруживаться и вызывать исключение.
  - Зависимость от несуществующего препромпта – в обычном режиме предупреждение, в строгом – ошибка.
  - Проверка порядка: зависимости должны добавляться перед требующим препромптом (если это важно). Можно тестировать порядок в итоговом списке.

### 4. Локализация заголовков (`test_localization.py`)
- Функция, получающая заголовок для категории с учётом заданных переводов.
- Проверить:
  - Если для языка `ru` задан перевод через `--translation=ru:roles-title:Роли` – возвращается "Роли".
  - Если перевода нет – возвращается стандартное название (например, "Roles").
  - Несколько переводов – последний имеет приоритет.
  - Регистр ключей не важен? Уточнить в ТЗ (вероятно, чувствителен, но можно сделать регистронезависимым для удобства).

### 5. Сборка финального документа (`test_assembler.py`)
- Функция `assemble_prompt(preprompts_dict, project_md_content, order, translations, lang)`.
- Проверить:
  - Порядок разделов соответствует `order`.
  - В каждом разделе текст препромптов объединяется в список.
  - Если в тексте препромпта есть markdown-список, он должен быть вложенным в общий список раздела (например, добавлен отступ).
  - Вставка `PROJECT.md` с понижением уровня заголовков: если в `PROJECT.md` есть `# Title`, в итоге должно стать `## Title` внутри раздела "Project Description" (или заданного заголовка).
  - Пустой `PROJECT.md` не вставляется.
  - Подсчёт токенов (если реализован) – проверка, что возвращается число.

### 6. Обработка конфигурационных файлов (`test_config.py`)
- Фикстура: временная директория с конфигами (`almai.options`, `almai.custom.options`, `~/.almai/.options`).
- Проверить:
  - Чтение опций из нескольких файлов с правильным приоритетом (основной → системный → пользовательский) и возможность отключения через `--no-*-options`.
  - Раскрытие макросов: `$(AppRoot)` заменяется на путь к корню приложения, `$(UserHome)` – на домашний каталог.
  - Поддержка комментариев и пустых строк.
  - Обработка `@`-файлов: если в командной строке указан `@file`, его содержимое добавляется как опции.
  - Вложенные `@`-файлы (если поддерживаются) – проверить рекурсивное включение.

### 7. CLI (`test_cli.py`)
- Использовать `subprocess` или `CliRunner` (если используется click). Рекомендую `subprocess` для эмуляции реального вызова.
- Проверить:
  - Основные опции: `--lang`, `--roles`, `--output` и т.д.
  - Множественные значения через запятую и через повторение опции.
  - Команда `--list` – вывод должен содержать идентификаторы и описания.
  - `--strict` и `--no-strict` – влияние на поведение (можно проверить на сценарии с отсутствующим файлом).
  - Обработка ошибок: при отсутствии preprompts и `PROJECT.md` программа должна завершиться с ошибкой.
  - Вывод в файл: проверить, что файл создан и содержит ожидаемый текст.

### 8. Обработка ошибок (`test_errors.py`)
- Проверить поведение в обычном и строгом режимах для каждой описанной в ТЗ ошибки (отсутствие файла, ошибка UTF-8, циклические зависимости и т.д.).
- Использовать `capsys` или `caplog` для проверки вывода предупреждений.

### 9. Интеграционные тесты (`test_integration.py`)
- Создать полноценную структуру проекта с preprompts, конфигами, `almai.yaml`, `PROJECT.md`.
- Запустить утилиту с различными комбинациями аргументов и проверить итоговый вывод на соответствие ожидаемому (можно сохранить ожидаемый результат в отдельном файле и сравнивать).
- Проверить сценарии:
  - Базовый запуск с preprompts из командной строки.
  - Запуск с использованием `project-preprompts` из `almai.yaml`.
  - Запуск с локализацией.
  - Запуск с зависимостями.

## Дополнительные указания
- Все тесты должны быть автономными: использовать временные директории (`tmp_path`) и не полагаться на внешние файлы.
- Для мокирования функций, работающих с файловой системой, можно использовать `monkeypatch` или `mocker`.
- Для проверки вывода в консоль использовать `capsys`.
- Ожидаемое поведение при ошибках: в обычном режиме программа продолжает работу (код возврата 0), в строгом – завершается с кодом 1 и сообщением в stderr.
- Не забыть про тесты на граничные случаи: пустые файлы, файлы только с пробелами, очень длинные строки и т.д.

## Пример оформления теста (иллюстрация)

```python
import pytest
from pathlib import Path
from almai.loader import load_preprompt  # гипотетическая функция

def test_load_preprompt_with_localization(tmp_path):
    # создаём структуру
    base = tmp_path / "preprompts"
    roles_en = base / "roles" / "en"
    roles_en.mkdir(parents=True)
    (roles_en / "cpp-dev.md").write_text("C++ developer (en)")
    roles_base = base / "roles"
    roles_base.mkdir()
    (roles_base / "cpp-dev.md").write_text("C++ developer (base)")

    search_paths = [base]
    # язык en - должна загрузиться en-версия
    content = load_preprompt("roles", "cpp-dev", lang="en", search_paths=search_paths)
    assert content == "C++ developer (en)"

    # язык ru - нет ru-версии, загрузится базовая
    content = load_preprompt("roles", "cpp-dev", lang="ru", search_paths=search_paths)
    assert content == "C++ developer (base)"
```
