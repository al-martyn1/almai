# Как описывать библиотеки для ИИ

Чтобы ИИ (большая языковая модель) смог работать с вашей закрытой библиотекой, нужно описать её **API-контракт**, а не исходный код. Главный принцип: **описание должно быть функциональным, а не энциклопедическим**.

Вот стратегия, как уместить описание в минимальный объем, сохранив полноту понимания для ИИ.

### 1. Ключевой принцип: Формат «Сигнатуры + Семантика»

ИИ лучше всего понимает структурированные данные. Используйте следующий шаблон:

1.  **Имя и Краткое Назначение** (1 предложение).
2.  **Список функций/классов** в формате, близком к коду (TypeScript-подобные сигнатуры или docstrings).
3.  **Важные побочные эффекты** (что функция *делает*, помимо возврата значения: пишет в файл, меняет глобальное состояние, делает сетевой запрос).

### 2. Конкретный формат описания

Вместо того чтобы описывать библиотеку прозой, используйте структуру «Псевдокод + Пояснения». Это самый компактный способ передать максимум смысла.

#### Шаг А: Определите «Точки входа»
Выберите только те 3-7 функций/классов, которые нужны для текущей задачи. Не описывайте всю библиотеку, если вам нужно использовать только её часть.

#### Шаг Б: Используйте сигнатуры с типами
ИИ отлично понимает TypeScript, Python type hints, или даже просто структурированные сигнатуры.

**Плохо (занимает много места и расплывчато):**
> «У нас есть функция для отправки данных. Она принимает словарь, который должен содержать ключ "user" и "message". Потом она проверяет, авторизован ли пользователь, используя глобальный токен, который устанавливается в отдельной функции инициации. Если авторизация не пройдена, она возвращает ошибку 403...»

**Хорошо (компактно и точно):**
```python
# Библиотека: PrivateAPIClient v2
# Инициализация: init(token: str) -> None
# Отправка: send(endpoint: str, payload: dict) -> dict
#   - Выбрасывает AuthError если init не был вызван
#   - Выбрасывает RateLimitError если >10 RPS
# Формат ответа: {"status": "ok", "data": any} | {"status": "error", "code": int}
```

### 3. Техника «Few-shot» (Обучение на примерах)

Вместо описания всех возможностей, покажите 1-2 примера использования в контексте промпта. ИИ экстраполирует логику на ваш запрос.

**Структура промпта:**
>Я использую закрытую библиотеку `MyLib`. Вот её API:
>[Вставьте компактное описание сигнатур и правил]
>
>Вот пример того, как я обычно использую её для получения данных:
>```python
>from mylib import Auth, Query
>
>client = Auth.login("key")
>result = Query.fetch("users", filters={"id": 1})
># result имеет поле .data и .metadata
>```
>
>**Задание:** Используя эту библиотеку, напиши функцию, которая обновляет пользователя, но только если он активен. Учитывай, что `update` возвращает `None` при ошибке валидации.

### 4. Используйте «Приоритизацию» (Core vs Extended)

Если библиотека большая, разделите описание на две части в промпте:
- **Core (обязательно):** То, что ИИ должен использовать (основные классы, инициализация).
- **Extended (справочно):** Если ИИ нужно что-то нестандартное, пусть спросит уточнение, либо дайте это отдельным блоком.

### 5. Технический прием: Сжатие через «Схему»

Если ваша библиотека работает с определенными объектами (например, `User`, `Transaction`), опишите только структуры данных в JSON Schema или Pydantic-подобном виде. Это занимает мало места, но задает жесткую логику.

```json
// Схемы объектов библиотеки (компактно)
User: {id: int, status: "active"|"blocked", meta: dict}
Response: {success: bool, data?: any, error?: string}
```

### Итоговый чек-лист для вашего промпта

Чтобы ИИ точно понял и не «галлюцинировал» (не придумывал методы из публичных библиотек):

1.  **Укажите контекст:** «Эта библиотека не публичная, не ищи её в своих данных, полагайся только на мое описание».
2.  **Дайте имена:** Укажите точные имена классов и функций.
3.  **Опишите исключения/ошибки:** Это критически важно, чтобы ИИ писал правильную обработку.
4.  **Запретите импорт неизвестного:** Скажите явно: *«Используй только те функции, которые я перечислил. Если нужно что-то еще — предположи, что это существует, но отметь это в комментарии TODO»*.

### Пример компактного описания (для отправки)

Если вы отправите ИИ что-то вроде этого, он сможет работать с библиотекой, даже никогда её не видев:

> **Контекст:** Есть приватная библиотека `SecureDB` (не в открытом доступе). Ниже её полное API, необходимое для задачи. Других методов нет.
>
> ```python
> # Инициализация
> SecureDB.init(connection_string: str) -> bool  # True если успех
>
> # Основные операции
> SecureDB.find(collection: str, query: dict) -> list[dict]
> SecureDB.insert_one(collection: str, doc: dict) -> str  # возвращает id
> SecureDB.update(collection: str, id: str, updates: dict) -> bool
>
> # Особенности:
> # 1. Все операции требуют предварительной инициализации.
> # 2. Если find не находит ничего, возвращает пустой список [], не None.
> # 3. update возвращает False, если документ с id не существует.
> ```
>
> **Задача:** Напиши функцию на Python, которая добавляет нового пользователя, но перед этим проверяет, нет ли пользователя с таким же email в коллекции "users". Используй методы выше.

Этот подход позволяет уложить описание API средней сложности в 10-20 строк текста, что составляет менее 500-1000 токенов, оставляя большую часть контекстного окна под решение вашей конкретной задачи.


# Примеры для C++

Для C++ подход похож, но нужно уделить внимание **управлению памятью**, **исключениям** и **шаблонам**, так как это критически важно для корректной генерации кода. ИИ должен понять, кто владеет ресурсами, как возвращаются объекты (по значению, по указателю, умные указатели) и какие ошибки возможны.

Ниже — несколько шаблонов описания от простого к сложному. Все примеры максимально компактны, но содержат всё необходимое.

---

## Пример 1. Простая C-подобная библиотека (функции, коды ошибок)

```cpp
// Библиотека: "ThermalControl" (приватная, работает с железом)
// Все функции возвращают 0 при успехе, иначе код ошибки (см. ErrCode).
// Инициализация обязательна перед любыми другими вызовами.

enum ErrCode {
    OK = 0,
    ERR_NOT_INIT = 1,
    ERR_OVERHEAT = 2,
    ERR_BUSY = 3
};

// Инициализация: открывает устройство, выделяет ресурсы.
int thermal_init(const char* port);

// Завершение: освобождает ресурсы.
void thermal_shutdown();

// Установка температуры (0..100). Блокирующая операция.
int thermal_set_temp(int celsius);

// Чтение текущей температуры.
int thermal_get_temp(int* out_celsius);

// Особенности:
// - thermal_get_temp заполняет out_celsius только при возврате OK.
// - Если вызвать thermal_set_temp до init, вернёт ERR_NOT_INIT.
```

**Почему этого достаточно:**  
ИИ видит сигнатуры, понимает, что память под `out_celsius` выделяет caller, знает возможные коды ошибок. Пример компактен, но описывает протокол.

---

## Пример 2. Класс с владением ресурсами (RAII, умные указатели)

```cpp
// Библиотека: "ImageProcessor" (закрытая)
// Все объекты — RAII. Исключения: std::runtime_error при ошибках.
// Основной класс:

class Image {
public:
    // Загружает изображение из файла. Бросает при ошибке.
    Image(const std::string& path);
    
    // Перемещение разрешено, копирование запрещено.
    Image(Image&& other);
    Image& operator=(Image&& other);
    
    // Методы:
    int width() const;
    int height() const;
    
    // Возвращает новый Image — результат применения фильтра.
    // Фильтры: "blur", "sharpen", "edges".
    Image apply_filter(const std::string& filter_name) const;
    
    // Сохраняет в файл. Бросает при ошибке.
    void save(const std::string& path) const;
};

// Вспомогательная функция:
std::vector<Image> load_batch(const std::vector<std::string>& paths);
```

**Почему этого достаточно:**  
Указаны правила копирования/перемещения (важно для C++), исключения, названия фильтров. ИИ не будет пытаться вызывать копирующий конструктор, если его нет. Увидев `apply_filter`, поймёт, что возвращается новый объект, а исходный не меняется.

## Пример 3. Шаблонная библиотека (контейнеры, итераторы)

```cpp
// Библиотека: "SimpleTree" (приватная)
// Двусвязное дерево, узлы хранят значения типа T.
// Итераторы — обычные указатели (легковесные).

template<typename T>
class Tree {
public:
    Tree();
    ~Tree();
    
    // Вставка: возвращает итератор на вставленный узел.
    // Если parent == end(), вставляет в корень (если корня нет — создаёт).
    iterator insert(iterator parent, const T& value);
    
    // Удаление узла и всех потомков.
    void erase(iterator pos);
    
    // Итераторы:
    iterator begin();   // первый в глубину
    iterator end();     // nullptr-подобный
};

// Особенности:
// - Итератор — это просто указатель на внутренний узел.
// - Разрешается сравнивать iterator с nullptr.
// - После erase итераторы на удалённые узлы инвалидируются.
```

**Почему этого достаточно:**  
ИИ видит шаблон, понимает семантику вставки (связь с parent), знает про инвалидацию итераторов. Без этого он мог бы написать опасный код.

## Пример 4. Библиотека с асинхронными вызовами (колбэки, std::function)

```cpp
// Библиотека: "AsyncDB" (приватная)
// Асинхронные запросы к базе данных. Все операции неблокирующие.
// Результат приходит через колбэк.

class DBClient {
public:
    // Инициализация с адресом сервера. Выбрасывает при ошибке подключения.
    explicit DBClient(const std::string& endpoint);
    
    // Асинхронный запрос. Колбэк вызывается ровно один раз.
    // Колбэк принимает (error_msg, result_json).
    // Если error_msg не пуст, result_json игнорировать.
    void query(const std::string& sql,
               std::function<void(std::string error, std::string result)> callback);
    
    // Отменяет все незавершённые запросы (деструктор делает то же самое).
    void cancel_all();
};
```

**Почему этого достаточно:**  
Указаны сигнатура колбэка, семантика вызова (ровно один раз), разделение ошибки и результата. ИИ сможет правильно обработать лямбды и lifecycle объекта.

## Пример 5. Описание через примеры (few-shot) + сигнатуры

Иногда эффективнее дать не только описания, но и 1–2 коротких примера использования. ИИ хорошо обобщает по аналогии.

```cpp
// Библиотека: "Cryptor" (закрытая)
// API:
//   std::vector<uint8_t> encrypt(const std::vector<uint8_t>& key,
//                                 const std::vector<uint8_t>& data);
//   std::vector<uint8_t> decrypt(const std::vector<uint8_t>& key,
//                                 const std::vector<uint8_t>& ciphertext);
// Бросает std::invalid_argument, если key пуст или data некорректна.

// Пример:
//   auto key = std::vector<uint8_t>{0x01, 0x02, 0x03};
//   auto plain = std::vector<uint8_t>{'H','i'};
//   auto encrypted = encrypt(key, plain);
//   auto decrypted = decrypt(key, encrypted);
//   // decrypted == plain

// Задача: написать функцию, которая шифрует строку и возвращает hex-строку.
```

**Почему это работает:**  
ИИ видит точные типы и пример вызова. Он не будет пытаться использовать `std::string` напрямую, если библиотека работает с `vector<uint8_t>`. Пример служит спецификацией.

## Общие рекомендации для C++ описаний

1. **Явно указывайте владение памятью:**  
   - Кто выделяет? (caller или callee)  
   - Возвращается ли сырой указатель, `std::unique_ptr`, `std::shared_ptr` или объект по значению?  
   - Если возвращается ссылка, укажите, на что она ссылается (не висячая ли).

2. **Опишите исключения:**  
   - Какие исключения могут быть выброшены? (тип, условия)  
   - Есть ли `noexcept` функции? Это важно для move-конструкторов и деструкторов.

3. **Сложные шаблоны:**  
   - Если есть SFINAE или концепты, можно кратко описать требования к типам, но часто достаточно показать примеры инстанцирования.

4. **Многопоточность (если есть):**  
   - Какие функции потокобезопасны?  
   - Есть ли мьютексы внутри?

5. **Используйте псевдокод с типами:**  
   ИИ хорошо понимает компактные сигнатуры в стиле `void func(int* out)` или `std::optional<Data> try_parse(...)`.

## Пример полного промпта для C++ (собираем всё вместе)

> У меня есть приватная библиотека `DeviceAPI` на C++. Она не входит в твои обучающие данные. Используй только то, что я опишу ниже.
> 
> ```cpp
> // DeviceAPI — работа с устройством через последовательный порт.
> // Все функции возвращают bool (true — успех). При false ошибка, код ошибки можно получить через last_error().
> 
> bool init(const char* port);                     // открыть порт
> void shutdown();                                 // закрыть порт
> bool send_command(uint8_t cmd, const uint8_t* data, size_t len); // отправить команду
> bool read_response(uint8_t* out_data, size_t* out_len);           // прочитать ответ (блокирующий)
> const char* last_error();                        // текст последней ошибки
> 
> // Особенности:
> // - Нельзя вызывать send_command до успешного init.
> // - read_response ждёт данные до 1 секунды, если таймаут — возвращает false.
> // - out_len должен указывать на переменную с размером буфера, при выходе содержит реально прочитанное.
> ```
> 
> **Задача:** Напиши функцию `bool set_parameter(uint8_t param_id, uint8_t value)`, которая формирует команду (cmd=0x10, data = [param_id, value]), отправляет, и проверяет, что устройство вернуло ACK (байт 0x06). Используй описанные функции, добавь проверку ошибок.

Такой промпт занимает меньше 1000 токенов, но даёт ИИ всё необходимое для генерации корректного кода на C++.

Если ваша библиотека очень большая, можно разбить описание на несколько таких блоков и давать их по мере необходимости в диалоге.


# Использование Doxygen

**doxygen-описание (или аналоги — Javadoc, XML-комментарии) — это идеальная основа** для компактного описания API для ИИ. Оно уже содержит структурированную информацию, которую модель может использовать, но её нужно «сжать» до сути, удалив избыточность и оставив только то, что важно для генерации кода.

Вот как извлечь максимум пользы из doxygen-комментариев, не перегружая контекст.

---

## 1. Что doxygen даёт готового

Хороший doxygen-блок уже содержит:

- **Типы параметров** (часто с именами)
- **Назначение параметров** (in/out)
- **Возвращаемое значение** (и его семантика)
- **Исключения** (`@throw`)
- **Пред- и постусловия** (`@pre`, `@post`)
- **Примечания по владению памятью** (`@note`)

Это именно то, что нужно ИИ, чтобы понять, как вызывать функцию. Проблема только в том, что doxygen часто многословен (есть описание алгоритма, примеры, история изменений), а контекстное окно ограничено.

---

## 2. Как сжать doxygen-описание для ИИ

### 2.1. Оставляйте только сигнатуру и ключевые теги

Из всего doxygen-блока для ИИ критичны:

- **Имя функции/класса**
- **Параметры** (с типами и признаком in/out, если не очевидно)
- **Возвращаемое значение** (и его смысл, особенно если это код ошибки, умный указатель, `std::optional` и т.п.)
- **Исключения** (`@throw`) — какие типы и при каких условиях
- **Краткая семантика** (что функция делает) — одно предложение
- **Важные ограничения** (например, «не потокобезопасно», «выделяет память, которую caller должен освободить через free_result»)

Всё остальное (подробное описание алгоритма, исторические заметки, длинные примеры) можно опустить.

### 2.2. Превращаем doxygen в компактную запись

**Исходный doxygen (C++):**
```cpp
/**
 * @brief Устанавливает температуру нагревателя.
 * 
 * Эта функция отправляет команду на устройство через последовательный порт.
 * Выполнение может занять до 5 секунд, так как устройству нужно время на нагрев.
 * 
 * @param celsius Температура в градусах Цельсия, от 0 до 100.
 * @return true если команда принята и температура достигнута.
 * @return false если произошла ошибка (см. get_last_error).
 * 
 * @throw std::invalid_argument если celsius вне диапазона.
 * @pre Инициализация устройства должна быть выполнена (init() вернула true).
 * @post После успешного вызова устройство поддерживает температуру в пределах ±2°C.
 * 
 * @note Функция блокирующая. Не вызывать из обработчиков прерываний.
 */
bool set_temperature(int celsius);
```

**Компактное описание для ИИ:**
```cpp
// Устанавливает температуру (0..100). Возвращает true при успехе, false при ошибке.
// Бросает std::invalid_argument, если celsius вне диапазона.
// Требует предварительной инициализации. Блокирующая.
bool set_temperature(int celsius);
```

Такой вариант занимает в 3–4 раза меньше токенов, но содержит всю необходимую информацию для генерации корректного вызова и обработки ошибок.

---

## 3. Автоматизация: парсинг doxygen в компактный формат

Если библиотека большая, можно написать небольшой скрипт, который извлекает из doxygen-комментариев только нужные поля и выводит их в сокращённом виде. Например, для C++ можно использовать:

- **Doxygen XML output** (через `GENERATE_XML = YES`) и парсить его XPath-запросами.
- **Clang tools** (например, `clang-query` или `libclang`) для извлечения аннотированных сущностей.
- Простой регулярный поиск по заголовочным файлам (если комментарии единообразны).

Результат можно представить в виде:

```
# Библиотека DeviceAPI (сокращённое описание)

## set_temperature(int celsius) -> bool
- Устанавливает температуру (0..100).
- Возвращает true при успехе.
- Бросает std::invalid_argument при выходе за диапазон.
- Требует инициализации.
```

Этот структурированный текст идеально подходит для вставки в промпт.

---

## 4. Особенности для C++: что doxygen часто недоговаривает

Даже с doxygen есть моменты, которые лучше добавить вручную, потому что они критичны для C++:

- **Владение памятью** (если функция возвращает сырой указатель — кто удаляет?).
- **Копируемость/перемещаемость** классов (особенно если doxygen не описывает специальные функции).
- **Потокобезопасность** (часто doxygen не указывает, но для ИИ важно).
- **Инвалидация итераторов/ссылок** (если есть контейнеры).

Эти вещи можно добавить в компактное описание как дополнительные маркеры.

## 5. Пример: преобразование doxygen-блока для целой библиотеки

Допустим, есть три функции с doxygen. Вместо вставки полных комментариев (которые могут занимать 50+ строк) можно дать сжатую таблицу.

**Полные doxygen (много места):**  
(каждый блок по 8-10 строк)

**Компактное описание:**
```cpp
// Библиотека: SensorReader (приватная)
// Все функции бросают std::runtime_error при ошибках связи.

/// Инициализация: порт, скорость. Возвращает true при успехе.
bool sensor_init(const char* port, int baudrate);

/// Чтение одного значения. Возвращает float, если успех.
/// Бросает std::invalid_argument, если сенсор отключён.
float read_sensor(int sensor_id);

/// Получить последнюю ошибку в виде строки.
const char* get_last_error();
```

ИИ получает всю суть, не отвлекаясь на описательные фразы.

## 6. Практический совет: подавайте doxygen как «справочный материал» отдельно

Если библиотека большая и вы не можете сократить всё в один промпт, можно сделать так:

1. **В первом сообщении** дать краткое описание (сигнатуры + ключевые особенности).
2. **Сказать ИИ:** «Если нужны детали по конкретной функции — спроси, я дам её полное doxygen-описание».

Так вы держите контекст лёгким, но при необходимости можете расширять.

## Итог

Doxygen-комментарии — **отличная база**, но напрямую вставлять их в промпт неэффективно. Используйте их как источник для извлечения:

- сигнатур,
- типов параметров,
- возвращаемых значений,
- исключений,
- краткой семантики.

Удаляйте всё, что не влияет на корректность вызова (подробные описания, историю, примеры). В результате вы получите компактное описание, которое позволяет ИИ писать правильный код, не перегружая контекстное окно.
